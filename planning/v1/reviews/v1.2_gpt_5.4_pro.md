  - yes, and this is one of the best implementation choices

  - yes
- **Sparse observations with masks**
  - yes
- **Python/JAX emission**
  - yes
- **Diagnostics and explainability**
  - yes, and better than minimum

## Exceeds the spec

You actually overshot v1 in a few productive ways:

- model/experiment separation in Python
- learned initial state support
- unit conversion emission for at least one dimension family
- soft penalty support (`smooth`)
- consistency policies (`off`, `equation_only`, `all`)
- JSON compile spec round-tripping
- JAX `lax.scan` backend rather than naive Python loops
- typed Python wrappers and explanation payloads

These are all healthy additions.

## Partial / mismatched against the spec

A few parts are only partially realized:

### A. Data binding is structural, not payload binding
In the spec, binding looked like it might include actual values and times. In the implementation, binding is mostly:

- role declaration
- schedule declaration
- interface declaration

and the actual forcing/observation values arrive at runtime in the emitted module.

I actually think this is a **healthy deviation**, but it means:

- compile-time validation can’t truly validate actual payload coverage/shapes
- runtime validation becomes essential

Right now runtime validation is thin.

### B. Constraints are only partially enforced
You parse bounds and penalties, but hard constraints are operationally enforced only in some places:

- slot outputs: projected
- learned initial states: projected
- equation outputs: not projected
- direct runtime inputs: not validated/projected
- state updates: not post-projected

That is only partial realization of the hard-constraint story.

### C. Compilation is not yet demand-driven
The spec implied compiling a plan for a particular requested workflow/output set. The current planner mostly tries to schedule everything it can in the single-step graph.

That’s fine for v1, but for real models it will become a problem.

### D. Branch/precondition semantics are not represented
The spec talked about exact/approximate, branch ambiguity, analytical preconditions. The code does not yet track those formally.

That is acceptable for toy arithmetic v1, but not for production science.

# 4. The biggest issues I would fix before widening scope

These are the load-bearing ones.

## 4.1 Critical: candidate provenance leakage through e-class extraction

This is the biggest issue in the current system.

### The problem
In `plan.rs`, equation candidates call `extract_available_expression(...)` using the candidate’s `output_class`.

But `output_class` is a global e-class for the output quantity, which may already contain expressions from **other relations** that also define that output.

So a candidate sourced from relation A can extract an expression that really came from relation B.

That means you can get:

- wrong provenance
- wrong source labels
- wrong costs
- misleading explanations
- corrupted alternative-path semantics

### Example failure mode
If two different relations define `transpiration`, then the candidate for `demand_transpiration` can resolve using the `supply_transpiration` expression if that one is currently available and cheaper.

Then the planner may say:

- source = `demand_transpiration`
- expression = supply path

That is wrong.

### Why it matters
This breaks one of the core promises of the spec:

- alternative paths should be explicit
- canonical path choice should be explainable
- provenance should remain meaningful

### Immediate recommendation
For v1.1/v2, I would **stop using whole-output e-class extraction as the candidate resolver**.

Instead, do one of these:

#### Option 1: easiest and probably best for now
For each directional candidate, resolve availability on the candidate’s own directional expression tree, with only **local rewrites / normalization** applied to that expression.

Use the e-graph only for:

- equivalence bookkeeping
- alternative path detection
- maybe commutative normalization

#### Option 2: harder but more “pure”
Keep candidate-local provenance tags and restrict extraction to the subspace reachable from that candidate’s seed expression.

That’s more elegant, but more work.

### What I’d do
I’d do **Option 1**.  
It is simpler and more correct for v2.

---

## 4.2 Major: planning is not demand-driven

Right now `build_single_step_plan` mostly forward-schedules whatever is reachable.

That works for the MVP, but for real models it will hurt you.

### Problems this causes
- you compute irrelevant quantities
- you can get false unresolveds/cycles from irrelevant subgraphs
- you may require bindings for states that the requested workflow does not actually need
- explanations get noisier
- compile cost grows with whole-model size instead of requested slice size

### What v2 should do
Make planning **demand-driven**.

Define a required set like:

- all `t+1` state lhs targets needed for rollout
- all observed quantities
- all explicitly requested outputs/save points
- any quantities needed by selected consistency terms

Then do:

1. backward slice to required quantities
2. choose candidate paths for only that slice
3. forward schedule the resulting acyclic subset

That will matter a lot once your plant model has 50+ variables.

---

## 4.3 Major: sparse `DataSeries` providers are not actually supported

The compile spec allows:

```rust
DirectBindingKind::DataSeries { steps: Vec<usize> }
```

But the emitted runtime code assumes forcing is simply present every step:

- `forcing["x"]`
- `forcing_series["x"]`

There is no missingness support for provider-side sparse data.

### So right now
- sparse observations: supported
- sparse direct providers: **not actually supported**

### Recommendation
For v1, I’d tighten the contract:

- `DataSeries` means **dense per-step forcing over the compiled horizon**

If you want sparse providers later, that is a different feature:
- interpolation
- imputation
- underdetermined state reconstruction
- masked forcing semantics

Do not pretend that is already there.

At minimum, change validation so that dense provider bindings must cover the full horizon in v1.

---

## 4.4 Major: hard constraints are only partially operational

This is the next thing I’d tighten.

### Currently
- slot outputs are projected
- learned initial states are projected
- direct inputs are not validated/projected
- mechanistic equation outputs are not projected/checked
- state updates are not post-projected

### Why this matters for your science
In plant models, some bounds are not decorative:

- stomatal conductance ≥ 0
- water potential often ≤ 0 in your chosen convention
- allocation fractions on simplex
- NSC pools ≥ 0
- phenology gates in [0,1]

If equation outputs or state updates violate these silently, the semantics get muddy fast.

### v2 recommendation
Make runtime semantics explicit by role:

- **provider outputs**: project if configured
- **learned states/params**: project if configured
- **mechanistic outputs**: either
  - assert,
  - project,
  - or add hard-fail diagnostics,
  depending on compile policy
- **state updates**: allow optional post-step projection/validation

You do not need a full inequality solver to do this.

---

## 4.5 Major: runtime input contract validation is too thin

This is the biggest practical gap between the compiler and real workflows.

Right now the compiled artifacts don’t strongly validate:

- required forcing keys
- required constant keys
- observation keys
- mask/value shape agreement
- horizon agreement
- slot provider arity contracts
- runtime dtype assumptions

For the tiny demo, that’s okay.

For real workflows, it will become frustrating.

### Recommendation
Add either:

- generated `validate_inputs(...)`
- or a Python wrapper that checks payloads against artifact metadata before rollout/loss

You already have most of the metadata. Use it.

---

## 4.6 Medium: constraints are still stringly and span-poor

`constraints.rs` is okay for v1, but it is now one of the first things that should become more structured.

Current limitations:

- parsed from raw strings
- only `self >= ...`, `self <= ...`, `penalty smooth(...)`
- no per-constraint spans
- no richer constraint AST

### Recommendation for v2
Parse constraints into a small AST with spans, e.g.:

- lower bound
- upper bound
- smooth penalty
- simplex
- maybe rate penalty / rate bound
- custom named constraint call

That will help both diagnostics and emission.

---

## 4.7 Medium: the math grammar is too narrow for the next real model

This is fine for v1, but it’s your next ceiling.

Missing for real physiology:

- scientific notation in expressions
- function calls
- `min` / `max` / smooth min
- exponentials/logs
- power laws
- piecewise/switch behavior
- maybe tuples/vectors

You do not need all of that at once. But you do need a plan.

---

## 4.8 Medium: the JAX backend is good for v1, not yet for scale

The generated JAX code works conceptually, but for bigger models it will hit limits:

- lots of Python dict/string-key access
- runtime slot providers as arbitrary Python callables
- full-history storage every rollout
- no save-point selection
- no batched/state-struct representation
- no JIT-friendly learned slot template contract

This is not a criticism of v1. Just don’t confuse it with the final runtime.

# 7. What v2 should be

My recommendation: **v2 should not be “more general Myco.”**  
It should be **“Myco that can express one real plant model family.”**

Specifically, v2 should unlock a first genuinely plant-relevant model.

## v2 goal
A discrete-time, differentiable, single-tree scientific model that can express:

- mechanistic relations
- trainable scalar parameters
- one learned controller slot
- sparse observations
- at least one implicit/algebraic subproblem or one-way registered function family

That would let you move toward Farquhar/Sperry territory without yet needing full continuous-time infrastructure.

---

## v2 feature priorities

## 7.1 Add a `parameter` quantity kind
Right now everything static is effectively a node + constant binding.

For real physiology, you want explicit parameters:
- `Vcmax`
- `Jmax`
- `P50`
- vulnerability shape
- capacitance
- respiration coefficients
- allometry constants

### Why this matters
Parameters are different from:
- external forcing
- state
- derived node

They are:
- static within a rollout
- often fit/trainable
- often trait-conditioned
- often hierarchical across species/sites

### v2 recommendation
Add a role like:

- `parameter`

and allow bindings like:
- fixed
- learned_scalar
- learned_vector
- maybe prior-backed later

This is essential for actual mechanistic model fitting.

---

## 7.2 Add a function registry
This is the biggest unlock for real science models.

You should not try to represent Farquhar, Sperry, Weibull curves, Arrhenius scaling, etc. only as primitive arithmetic in surface syntax.

### Add a registry concept
A function definition should carry:
- signature
- dimension/kind contract
- backend emission implementation
- optional directional implementations
- exact/approximate flag
- optional preconditions
- optional differentiability notes

### Example categories
- forward-only function
- forward + analytic inverse
- forward + monotone numerical inverse
- forward-only but differentiable
- piecewise but smooth surrogate available

### This is how you will handle:
- Farquhar photosynthesis pieces
- vulnerability curves
- temperature response functions
- saturating uptake curves
- softmin/smooth thresholds

This should be a v2 priority.

---

## 7.3 Add local solve blocks / SCC solve support
This is the other huge unlock.

For real plant models, same-step loops are not rare. They are standard.

Examples:
- \(A\), \(C_i\), \(g_s\) coupling
- hydraulic supply-demand balance
- canopy \(\psi\) and transpiration equilibrium

Your current v1 correctly rejects algebraic loops.

For v2/final, you need a way to say:

> this strongly connected local subsystem is to be solved numerically each step

### Suggested design
Add a construct like a `solve` block or implicit subproblem block:

- named residual equations
- declared unknowns
- declared inputs/parameters
- solver hint
- backend emitter hook

This is much better than trying to make general symbolic inversion solve everything.

---

## 7.4 Make planning demand-driven
I mentioned this above, but in v2 it becomes essential.

Required set should be driven by:
- temporal state updates
- observations
- explicit requested outputs/save vars
- requested consistency terms

This will make the compiler usable on bigger models.

---

## 7.5 Add observation operators
This is crucial for your actual research workflows.

Right now observations are identity-on-node.

Real plant workflows need operators like:

- sap flux from transpiration and sapwood area
- gas exchange operator
- predawn/midday sampling operator
- aggregated canopy/plot variables
- stand-level sums
- destructive sample operators

### v2 recommendation
Add at least:
- identity operator
- affine/simple expression operator
- named registry-backed observation operator

And add observation metadata:
- loss kind
- weight / sigma
- mask
- maybe aggregation axis later

---

## 7.6 Improve JAX backend representation
For v2 I would start moving away from string-key dicts as the only backend representation.

You don’t need to fully abandon readable emitted source, but consider:

- static field ordering
- lightweight state/forcing structs
- save selection
- batched rollout support

For production JAX training, you’ll want something closer to:
- struct-of-arrays
- fixed slot interface
- fewer Python dictionary lookups inside traced code

# 9. How I would map Farquhar / Sperry / Potkay into the final system

## Farquhar-style workflows
You will need:

- function registry
- piecewise or smooth-limitation support
- local same-step solve for \(A\) and \(C_i\)

A likely pattern:

- `A = farquhar(ci, vcmax, jmax, temp, par, ...)`
- `A = gs_co2 * (ca - ci)`
- `E = gs_h2o * vpd`
- maybe `gs_h2o = 1.6 * gs_co2`

This is a local implicit solve block, not just directed primitive arithmetic.

---

## Sperry-style workflows
You will need:

- hydraulic vulnerability functions
- monotone inverses or local root solves
- supply-demand coupling
- maybe canopy water potential equilibrium

Likely pattern:

- `E = hydraulic_supply(psi_soil, psi_leaf, vuln_params, kmax, ...)`
- `E = transpiration_demand(gs, vpd)`
- maybe `cost = hydraulic_cost(psi_leaf, vuln_curve)`

Again, solve block territory.

---

## Potkay-style workflows
You will need:

- explicit state pools
- discrete temporal updates
- allocation fractions with simplex constraints
- growth/respiration bookkeeping
- perhaps a learned controller replacing the objective/heuristic

This maps very naturally to:
- `parameter`
- `state`
- `slot controller`
- `simplex` and bound constraints
- observation operators

This is probably the easiest of the three to prototype in Myco after you have discrete parameters and constraints.

## Phase E: only then consider continuous-time/stiff solver support
By the time you actually need full continuous-time differentiable hydraulics, you’ll know which parts deserve first-class support.

# 11. One very important strategic point

For your actual research workflows, I would **not** rush Myco straight to full continuous-time Neural-ODE/UDE land.

You can get very far with a **hybrid discrete-time system**:

- hourly or half-hourly time grid
- within-step mechanistic algebraic solves
- temporal state updates for water / carbon / allocation
- sparse observation masks
- learned controller slot

That is already enough to test a lot of the central scientific idea:
- controller over mechanics
- partial observation learning
- biological constraints
- learned non-teleological strategy

That path is much more tractable than going immediately to stiff adjoint ODE infrastructure.
