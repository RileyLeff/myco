- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §7 — faithfulness × orientation 2×3 matrix.** The chunk report commits to two orthogonal axes (faithfulness: lossless / lossy-model / lossy-tolerance; orientation: bidirectional / unidirectional) with a 2×3 matrix of representative examples. §17 preamble cites the 2×3 faithfulness × orientation matrix as a defining structure of the section. The representative examples (`convert`, `identify`, `approximate`, relation `=`) match those in §17.2 and the preamble.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §12 (CC5) — unified rewrite-predicate language.** CC5 locks that rewrite predicates unify value-bound envelope predicates and structural site-scoped predicates under one predicate language with no new rewrite category. §17.4 absorbs this as the unified predicate language covering refinements, capability satisfaction, structural shape, and unit algebra.

- **`planning/v2/spec_dev_notes.md` (2026-04-20 functions lock) — function-inverse rewrites fire from stdlib-declared capability contracts, not user annotations.** The dev notes record Option B: stdlib atoms carry `Invertible<_>`, `Differentiable`, `Monotone` contracts; user functions have no property-declaration surface; compiler derives from body composition. §17.3 absorbs this decision verbatim, including the user's only recourse being to refactor into composable stdlib pieces.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §11 table row 15 and O2.2 appendix — rewrite groups A-Y with ~58 rules.** The chunk report commits to 25 lettered groups (A-Y) with approximately 60 rules organized by natural category. §17.5 absorbs the group naming and the four representative group descriptions (A algebraic, E equality/merge, Y closure-policy, Z distribution-family). The full catalog is deferred to Appendix C, matching the chunk report's acknowledgement that the full catalog belongs in an appendix.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §11 rows 15 and 29 — default-on / default-off partition.** O2.2's Bucket 1 (fully committed, no blocking dependencies) maps to §17.6's default-on set; O2.3's approximate-tier surface maps to default-off. §17.6 absorbs this as the baseline rewrite partition, including the consequence that a model compiles with zero authorized approximations if no `approximate` block is written.

- **`planning/v2/anti_spec.md` — `#[inverse]` annotation retired; four-class invertibility metadata retired.** Anti-spec records retirement of `#[inverse]`, `bijective` / `injective_restricted` / `lossy` / `opaque` four-class metadata, and `#[...]` attribute annotations generally. §17.3 reflects the retirement by stating the user has no annotation path to declare a function invertible; §17.6 default-on set fires stdlib-declared inverses without user annotation.

- **`planning/v2/spec_dev_notes.md` — §17 merge-sources one-liner updated to qualify function inverses as stdlib-declared.** The dev notes record that after the functions-section lock the §17 one-liner was patched. The patched text matches the §17.1 treatment of source 5 (stdlib-declared function inverses via capability contracts on fns; see §6).

- **`planning/v2/open_questions.md` (Tier 0 audit) — `~` is Layer 2 distributional metadata, not a merge source.** The audit section records that `~` is not an e-graph merge and is not in the eight-source list. §13.9 (cross-referenced in §17 preamble) and the §17 eight-source enumeration both reflect this: `observe` writes layer 2 envelope metadata, not layer 1 merges.

## Superseded

- **`planning/v2/spec.md` §8.1 — four-class invertibility metadata (`bijective`, `injective_restricted`, `lossy`, `opaque`) as user-declared operation properties.** The old spec treats these as user-facing declarations on both stdlib and user-registered functions (with an explicit user-facing `fn vulnerability_curve` example at lines 2098-2113 carrying `invertibility: injective_restricted`). §17.3 supersedes this: the user has no annotation path; the compiler derives properties from body composition and stdlib atom contracts; user functions have no property-declaration surface. **Already in anti_spec.md** ("four-class invertibility metadata (`bijective` / `injective_restricted` / `lossy` / `opaque`) | capability contracts on fns | same").

- **`planning/v2/spec.md` §9.2-9.3 — user-declared inverse functions with explicit `fn inverse vulnerability_curve(...)` syntax and `#[verified_externally]` escape hatch.** The old spec allows users to write an explicit inverse as a named function, verified by symbolic check plus round-trip sanity, falling back to `#[verified_externally]` if only sampling-based. §17.3 supersedes this: no user annotation path exists; composition is the only escape hatch. **Already in anti_spec.md** ("`#[inverse]` | capability contract (`Invertible<_>`) on stdlib fn | unified contract machinery" and "`#[verified_externally]` | nothing | no proof-escape-hatch annotations").

- **`planning/v2/spec.md` §8.5, line 2074 — "The metadata is available to both standard library closure policies and user-defined ones via structural introspection on the competing paths."** Structural introspection on competing paths is retired. §17 (via §17.4's predicate language and §8.7's closure policy interface) supersedes this: custom closure policies receive only candidate values and user hyperparameters; no structural introspection surface. The `open_questions.md` Spec Maintenance section also flags this phrase for replacement. **Already in anti_spec.md** ("structural introspection (`<:` predicate, §5.5/§8.5) | nothing | closure policies see values + hyperparameters only").

- **`planning/v2/spec.md` §14.6 — `condition_weighted` listed as a shipping stdlib closure policy.** The old spec lists `condition_weighted` alongside `weighted_average`, `soft_select`, and `hard_select` as a standard library closure policy. ## Homeless

Corpus content that is relevant to §17, is not accounted for in spec_new.md §17, and is not already committed to anti_spec.md. This is the highest-value bucket.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §7 — envelope-narrowing interaction with faithfulness.** The chunk report states: *"A normally-lossy rewrite can become lossless in context when envelope metadata (Layer 2) proves the value is in a regime where the loss vanishes."* This principle is named and illustrated (Float64 → Float32 becoming lossless when the envelope proves `value ∈ [0, 1]`). §17.6 identifies default-on (lossless) and default-off (lossy) buckets but does not state that a default-off rewrite can be promoted to default-on contextually when the envelope narrows the loss to zero. The promotion rule is load-bearing for extraction correctness: without it, the compiler must conservatively treat all approximate rewrites as default-off even when they degenerate to lossless in context.
  **Recommend:** Add to §17.6 as a named corollary — something like "A default-off rewrite whose declared `error_bound` evaluates to zero over the e-class envelope is treated as default-on for that class; the promotion is diagnostic-reportable." One paragraph; no new machinery required.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §6, item 3 — algebraic rewrites are "compiler-supplied" from "a baseline rewrite set committed in the spec; users can extend via workflow configuration."** The chunk report notes that the baseline rewrite set is committed and that users may extend via workflow configuration (O2.2 open sub-question 5/6 pending, but the extension principle itself is stated). §17.3 and §17.4 do not say whether users can register additional rewrite predicates or whether the set is fixed by the spec. The phrasing in §17.4 ("compile-time only; runtime-observed values do not drive rewrites") answers a different question (no runtime-value guards) and does not address user extension.
  **Recommend:** Add a one-sentence clarification to §17.4 or §17.5 stating that the A-Y catalog (Appendix C) is the committed baseline and that workflow-level extension is tracked as an open item in §35 Other Opens. This prevents reader confusion about whether the rewrite set is closed.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §11 (O4.1) — `replaces` obligation retraction and its interaction with merge monotonicity.** The chunk report marks O4.1 as a Section 12 open: the `replaces` keyword implies deletion semantics that conflict with the e-graph's monotonic append-only invariant. Three options are enumerated; none is locked. §17 establishes the monotonicity invariant as a property of the equational core (via the eight-source enumeration, which only adds merges) but does not acknowledge the open tension with `replaces`. A reader of §17.1 could incorrectly infer that `replaces` is simply another merge source when it is not.
  **Recommend:** Add a parenthetical or forward-reference note in §17.1 (or in §16.2 which owns the monotonicity invariant) stating that `replaces` is a suppression mechanism, not a retraction — new facts suppress prior generation; prior e-classes are not deleted. Cross-reference §34 for the open design on arbitrary prior-claim retraction. This is a stable framing decision (the referential-truth / non-retraction stance is locked in chunk 04 §5); it just has not propagated into §17's prose.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §11 table row 28 (O4.2 resolved) — `identify`-seam merges as site-scoped structural-predicate-gated rewrites (CC5).** The chunk report resolves O4.2 by classifying `identify`-seam merges under the unified predicate language as site-scoped rewrites. §17.2's description of `identify` says it "merges their e-classes" and is "idempotent" but does not mention that at mesh nodes coincident with a declared seam (e.g., `identify phi=0 <-> phi=2*pi`), the merge fires as a structural-predicate-gated rewrite rather than an unconditional merge. Without this note, §17.2 implies `identify` always fires unconditionally, which is not true for geometry-scoped seam forms.
  **Recommend:** Add a note to §17.2 distinguishing module-scope `identify x = y` (unconditional merge) from geometry-body `identify coord_a <-> coord_b` (site-scoped structural-predicate-gated rewrite firing at coincident mesh nodes per CC5). One sentence; cross-reference §11 geometry.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` O4.3 — per-residual exposure for training emission requires extraction to preserve original relation names through canonicalization.** This is an extraction-constraint that bears directly on which rewrites in §17 can fire safely: algebraic rewrites (source 3, A-group) that canonicalize or CSE-collapse relations would destroy the per-residual names the training emission requires. The constraint is tracked as O4.3 (Section 12 open, no change). §17 does not acknowledge it. A reader extending the A-group rewrite set could unknowingly write rules that invalidate training emission.
  **Recommend:** Add a forward-reference note in §17.6 (the partition section) stating that default-on rewrites that collapse structurally distinct but algebraically equivalent relations (e.g., CSE-style canonicalization) must respect the per-residual exposure constraint; cross-reference §25 Training Emission. Alternatively, open a new `open_questions.md` entry under Tier 1 — Compiler/Runtime documenting the interaction. Either path prevents silent correctness bugs during implementation.

- **`planning/v2/v2.1_chunk_reports/04_egraph_foundation_in_progress.md` §11 (O4.6) — heterogeneous `argmax` tagged handles as a novel e-graph frontier.** The chunk report flags that heterogeneous `argmax` with `(pool_id, index)` tagged handles requires a new kind of e-node (runtime sum type) that cannot be compile-time merged across branches. This intersects §17 because it is an exception to the eight-source enumeration: tagged handles do not merge via any of the eight sources; they require a distinct e-node kind. O4.6 is open (Section 12 open). §17 does not acknowledge the carve-out.
  **Recommend:** This is legitimate in-progress design work (chunk 04 is an open chunk). Do not add to §17 until O4.6 is resolved. Open an `open_questions.md` entry if one does not already exist. Not homeless in the sense of a missed stable decision, just unresolved design that §17 will eventually need to address. No §17 action needed now.

- **`planning/v2/v2.1_chunk_reports/03_kernels_in_progress.md` §7 — three-tier optimization cut (lossless / lossy-as-model-claim / lossy-as-tolerance) and the observation that only lossless had syntactic surface at the time.** The kernel report surfaced the gap that drove chunk 04's `approximate` block design. The gap is now closed (§17.6 names the default-off set, and §15 owns the `approximate` block surface). No residual homeless content.

- **`planning/v2/spec.md` §9.5 — `deriv` and `integrate` as compiler primitives with e-graph consequences.** The old spec describes `deriv` as an expression-level primitive ("not runtime functions — they are..."). These participate in algebraic rewrites (A-group includes associativity / distributivity etc.; integration-by-parts is named in spec_dev_notes as an e-graph rewrite in §14.3). §17.1 does not call out symbolic-math primitives (`deriv`, `integrate`) as a merge source or as a rewrite family with its own group letter. The A-Y catalog lists them (the chunk 04 appendix includes a rule for integration-by-parts under the rewrite groups), but §17 prose does not forward-reference them. This is thin prose coverage for a non-trivial merge-adjacent mechanism.
  **Recommend:** Add a sentence to §17.4 or §17.5 noting that symbolic-math intrinsics (`deriv`, `integrate`) participate in the rewrite system via their own group entries in Appendix C and share the unified predicate language. This prevents a reader from concluding the rewrite system only covers arithmetic algebra.

- **`planning/v2/open_questions.md` Tier 0 — "When do two expressions get merged into one e-class?" identified as the single biggest unstated semantic.** The audit section flags that the spec relies on "merging happened somehow" without stating the trigger precisely (syntactic identity? after unit/convert normalization? only after user-declared equivalence?). §17.1 partially addresses this by enumerating the eight sources, but does not state the negative: merges happen *only* via the eight sources, never by silent inference beyond compiler-known rewrites. The Tier 0 entry proposes the answer: syntactic identity + compiler-internal rewrites (constant folding, associativity, convert normalization) + user-declared equalities via `=`, no silent inference beyond compiler-known rewrites.
  **Recommend:** Add a one-sentence negative to §17.1: "Merges are introduced only by the eight sources above; the compiler does not silently infer equalities beyond compiler-known rewrites." This closes the biggest semantic ambiguity identified in the Tier 0 audit and costs one sentence.

## Conflicts

Direct contradictions between spec_new.md §17 and any corpus document.

- **Source 2 naming: §17.1 calls it "workflow constant injection" but chunk 04 §6 item 2 calls it "observations injected at workflow time" and gives `provider.bind("y[2]", 5.3)` as the canonical example.** The chunk report's framing treats workflow-time value injection and observe-style data injection as one source. §17.1 explicitly separates them: source 2 is workflow constant injection (via `assume_constant`, `bind_known_constants`, and related verbs), which collapses a model variable with a literal value; `observe` (§13.8) writes layer 2 envelope metadata and is not a merge source. The §17 preamble "terminology note" explains the split.
  **Recommend:** The §17.1 framing is the authoritative one (the split is a stable post-chunk-04 decision recorded in `spec_dev_notes.md`). Chunk 04 §6 is an in-progress document and predates the renaming; no action needed on chunk 04. When chunk 04's text is written up into spec normative content, item 2 must be updated to the §17.1 naming and scope.

- **Function-inverse user annotation: `planning/v2/spec.md` §9.2 provides `fn vulnerability_curve` with explicit user-declared `invertibility: injective_restricted` and a companion `fn inverse vulnerability_curve(...)` block, treated as the standard pattern. §17.3 states the user has no annotation path to declare a function invertible.** These are a direct contradiction. The old spec §9.2 pattern is not an edge case; it is the feature's worked example.
  **Recommend:** The §17.3 (and anti_spec.md) framing is authoritative. The spec.md §9.2 worked example must not be imported into spec_new.md. The `vulnerability_curve` mock needs rewriting per anti_spec.md's note that `mock_sperry.myco` uses `property monotone: ...` (which is in the same retired family). Any audit of §9 in spec_new.md should flag §9.2's pattern as superseded by §6 (functions, Option B) and §17.3.

- **`planning/v2/spec.md` §14.6 lists `condition_weighted` as a deferred stdlib closure policy. `planning/v2/open_questions.md` Spec Maintenance section says "Drop `condition_weighted` from the v2.1 stdlib policy list. Note it as a deferred compiler intrinsic." Chunk 04 O4.5 un-defers it and ships it as Y4. §17.5 references the Y-group as including Y1-Y6 closure policies.** The contradiction is between spec.md §14.6 (deferred), open_questions.md Spec Maintenance (deferred), and chunk 04 O4.5 / §17.5 (shipping). The chunk 04 O4.5 resolution is the authoritative one, locked after the other two documents.
  **Recommend:** The Y4 / `condition_weighted` un-deferral in chunk 04 O4.5 is the correct current state. The Spec Maintenance entry in `open_questions.md` predates it and should be marked resolved. No change to §17.5 needed. When §8.7 closure policies are written up in spec_new.md, the policy list must include Y4 `condition_weighted`.
