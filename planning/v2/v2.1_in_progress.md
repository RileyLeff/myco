# Myco v2.1 Language Features — Status & Open Questions

This document tracks every language feature, its status (settled, in progress, or
undesigned), and what remains to pin down. Features from the original v2 spec are
included for completeness; new features from the April 2026 design sessions are
marked as such.

---

## Module System

### `module path::name`
Module declaration. Provides namespacing, becomes the basis for imports.
**Status: settled.**

### `use path::{items}`
Imports. Modules may re-export.
**Status: settled.**

### `pub`
Inter-module visibility. Items private by default. `pub` controls inter-module
visibility only — workflow layer can bind any path regardless.
**Status: settled.**

### Library vs model modules
Libraries define reusable components (types, contracts, functions). Model modules
instantiate with `node`. Like Rust lib vs bin.
**Status: settled in concept.** Open: add the lib/bin analogy framing to the spec
prose (tracked in riley_spec_notes.txt).

## Type System

### `type Foo : Bar` — newtype / scalar alias
Creates a named type wrapping another. `type FishMass : Scalar<kg>`. Used for
semantic type safety — `FishMass + WaterMass` is a compile error even though both
are `kg`.
**Status: settled.** Open: clarify that "atomic" means leaf of the containment
tree (holds a numerical value), not "single-field."

### `type Foo { fields }` — composite type
Composite structural definition. Contains fields, constraints, relations.
**Status: settled.**

### `node name: Type` — instantiation
Creates a concrete instance in a model module. `node tree: SperryTree<...>`.
Reserved for model modules only — library modules use `type` exclusively.
**Status: settled.**

### Named types and conservation groups
Same-dimension types with different names are incompatible. `FishMass` and
`WaterMass` cannot be added. Named types form conservation groups via type
hierarchy:
```
type Mass : Scalar<kg> { conserved }
type FishMass : Mass
type DetritusMass : Mass
type PlanktonMass : Mass
```
The parent type (`Mass`) declares the conserved quantity and its units. Sibling
named types are all variants of the same underlying substance. Cross-sibling
arithmetic is only legal through explicit `convert` declarations. `{ conserved }`
on the parent activates compiler balance checking at rule boundaries.

Non-conserved parents work the same way for type grouping but without balance
checking:
```
type Conductance : Scalar<mol_per_m2_s_MPa>
type StemConductance : Conductance
type LeafConductance : Conductance
```

**Conservation is dimension-dependent.** The user must choose units where the
conservation law is naturally expressed. Carbon is conserved in moles (1 mol C
→ 1 mol CO2), not in kg (1 kg C → 3.67 kg CO2, because oxygen mass enters).
The compiler checks balance in the units declared on the conserved parent.
**Status: settled.**

### `convert` — named-type conversion
Two forms: bidirectional (`<->`) and one-way (`->`).

**Bidirectional** — the common case. Body is a relation (acausal equality).
Compiler can solve in either direction. Must be invertible:
```
convert (e: Euros) <-> (d: Dollars) {
    universal rate: Scalar<dimensionless> = 1.08
    d = e * rate
}
```

**One-way** — for genuinely non-invertible conversions. Compiler can only
solve forward:
```
convert Plaintext -> Ciphertext { ... }
```

**Bare** — 1:1 relabel between siblings of the same conserved parent. No body
needed. Same magnitude, different named type:
```
convert FishMass <-> DetritusMass
```

**Parameterized** — free quantities in the body are resolved from context at
the use site, like any other graph quantity:
```
convert (w: WetBiomass) <-> (d: DryBiomass) {
    moisture: Scalar<fraction>
    d = w * (1.0 - moisture)
}
```

**At the call site:**
```
waste.mass = convert(0.3 * prey.mass)       // compiler resolves types
dry_mass = convert(wet_sample)              // parameterized, moisture from graph
```

**Units flow through convert bodies** the same way they flow through any
relation — the compiler does dimensional analysis on the body. Bare converts
require the same dimension on both sides. Converts with bodies can cross
dimensions if the math checks out (e.g., moles → grams via molar mass).

**`->` is the primitive; `<->` is sugar** for both directions plus an
invertibility proof. If the body of a `<->` convert isn't invertible, compile
error.

**`where` preconditions** work on converts. The three-way overdetermination
classification applies: provably true → elide check, provably false → compile
error, undecidable → runtime assertion.
**Status: new, settled.**

## Generics

### Type parameters — `<V: VulnerabilityCurve>`
Types and functions can be generic over contracts.
**Status: settled.**

### Value parameters — `<N_SOIL: val>`
Types can be parameterized by compile-time-known positive integers. Keyword is
`val` (not `const` or `usize` — those scare scientists). `val` signals "this is
a value, not a type" without leaking implementation details.
**Status: settled (updated from `const N: usize`).**

### `where` clauses — `where N >= 2`
Bound verification on const generics.
**Status: settled.**

### Named generic arguments — required when 2+ parameters
Generic instantiation syntax depends on the number of parameters on the
target type:
- **Exactly one parameter:** positional is allowed. `Scalar<kg>`,
  `XylemSegment<WeibullVC>`. Named form is also legal
  (`Scalar<U = kg>`) but redundant for the single-parameter case.
- **Two or more parameters:** named is required. `SperryTree<V = WeibullVC,
  P = FarquharC3, N_SOIL = 4, N_CANOPY = 2>`. No positional form, no
  partial-positional. Argument-order memorization is a maintenance trap.

Rationale: single-parameter positional reads like mathematical notation
("a scalar in kg") and has exactly one unambiguous slot; multi-parameter
named is self-documenting and survives parameter-order refactors. The
passthrough case `fn arrhenius<U: Unit>(x: Scalar<U>) -> Scalar<U>`
falls out — positional binding to the local generic.

**API-break note:** adding a second generic parameter to a previously
single-parameter type will break all existing positional uses at compile
time. This is intended — adding a generic parameter is already an API
break, and failing loudly at every call site is the correct behavior.
**Status: settled.** Needs spec prose.

### ~~`dyn Contract`~~ → `impl Contract` — heterogeneous collections
`[PFT<impl Photosynthesis>; N]` in library modules. Users specialize per-element
in model modules. Compiler monomorphizes — `impl` is gone before flattening. No
runtime dispatch. Renamed from `dyn` to avoid collision with `some` (runtime
variable sizing).
**Status: settled (renamed).**

## Contracts

### `contract Name { ... }`
Trait/interface with inputs and outputs. The hybrid design: non-input members
(parameters like `b`, `c`) are real graph quantities. Input members are formal
parameters defining the invocation signature.
**Status: settled.**

### `input` / `output`
Declare the invocation signature of a contract. Inputs are functional parameters,
outputs are structural obligations.
**Status: settled.**

### Contract defaults
Default relation implementations. Included if and only if the implementing type
doesn't provide its own relation for that output.
**Status: settled.**

### Contract satisfaction
Two parts: input signature (functional) and output obligation (structural).
**Status: settled.**

### Data contracts (output-only)
A contract may have zero inputs. Such a contract declares a pure field-set
requirement — "to satisfy me, expose these fields with these types." This is
the mechanism that handles the shared-controller portability problem: a
`Tree` data contract enumerates the interface fields a controller consumes,
and any concrete model that exposes those fields via its own internal
relations satisfies the contract, regardless of internal structure
(`N_SOIL`, VC choice, species). Different studies with different concrete
implementations all plug into the same controller because they all satisfy
the same data contract.
**Status: settled.**

### Multi-contract satisfaction — `: A + B + C`
A concrete type may satisfy multiple contracts. Syntax is Rust-style `+`
separator:

```myco
pub type SperryTree<V, P, N_SOIL, N_CANOPY> : Tree + Observable + Simulated {
    // ...
}
```

Comma-separated would collide with generic bounds; `+` is unambiguous. Each
contract's field obligations compose — the concrete type must satisfy all of
them. Same-named fields across contracts must agree on type (compile error
otherwise).
**Status: settled.**

### Supertraits — `contract B : A + ...`
A contract may extend other contracts. Any type satisfying `B` automatically
satisfies `A`. The extending contract may add new field obligations.

```myco
pub contract Tree : Observable + Simulated {
    output interface_leaf_water_potential: WaterPotential
    output interface_vpd: Pressure
    // ...
}
```

Diamond inheritance is fine because contracts carry no implementation — if
two ancestors declare the same-named field with the same type, it's a
no-op; different types on the same name is a compile error.
**Status: settled.**

### Contract composition alias — `contract X = A + B`
Pure aggregation with no new fields is an ergonomic alias. `contract PlantModel
= Tree + CarbonBudget` just names the union.
**Status: provisional — include if it proves useful, defer otherwise.**

### Parameterized contracts
Contracts may be generic over other contracts:
```myco
contract TreeWithController<C: Controller> : Tree { ... }
```
Falls out of existing generics × contracts. Not load-bearing for v2.1 but
syntactically available.
**Status: settled as a consequence of generics + contracts.**

## Functions

### `fn name<generics>(params) -> ReturnType { body }`
Registered functions with typed parameters. Compiler uses these for dimensional
analysis, symbolic differentiation, and solver emission.
**Status: settled.**

### Function-level generics
`fn arrhenius<U: Unit>(value_25: Scalar<U>, ...) -> Scalar<U>`. Same
monomorphization as types.
**Status: settled.**

### Annotation blocks
`invertibility: bijective`, `differentiability: smooth`. Declared per-function.
Compiler uses these for symbolic analysis.
**Status: settled.**

## Relations & Equations

### `relation name: equation`
Acausal equality. `=` means equality, not assignment. Compiler solves in either
direction. Relations connect quantities across types.
**Status: settled.**

### `relation name for i in 0..N:` — index-style iteration
Iteration over fixed-size arrays. Generates one equation per element.
**Status: settled.**

### `for x in collection` — iterator-style iteration
Extends quantification to `some`-sized collections and graph neighborhoods.
Same declarative semantics as `for i in 0..N` — "for each x, this relation
holds." Works at relation-level and nested within relation bodies.
```
// Dynamic collection:
relation climate for t in trees:
    t.ambient_temp = air_temp.at(t.x, t.y)

// Graph neighborhood:
relation pit_drop on junction:
    for c in children:
        trace(pressure, edge = parent) - trace(pressure, edge = c)
            = pit_resistance * trace(axial_flux, edge = c)

// Filtering — value-based and type-based:
stressed_c3 = sum(t.lai for t in trees
    where t.photo is FarquharC3 and t.water < threshold)
```
Type-based `where` narrows the type within scope — fields specific to the
concrete type become accessible. Both index-style and iterator-style coexist;
different situations call for different forms.

Full design in `planning/v2/v2.1_chunk_reports/02_collections_iteration_report.md`.
**Status: new, settled.**

### `temporal name: d(x) = expr` / `step(x) = expr`
Two surface forms: `d(x)` declares a continuous rate (compiler owns
integration); `step(x)` declares a discrete update (compiler applies RHS
verbatim each tick, RHS reads prior values, LHS writes current). Use `d(·)`
for ODEs, `step(·)` for ratchets/accumulators/counters. `dt` is workflow-
layer cadence for `step(·)`; not referenced in `d(·)` RHS.
**Status: settled.**

### `initial name: equation`
Declares an algebraic relationship that must hold at t=0 for a temporal
quantity whose initial value is determined by other quantities in the graph
rather than supplied externally. Complements the workflow-layer
`assume_initial` / `learn_initial` / `learn_trajectory` verbs, which are
mutually exclusive per quantity. Example: `min_historical_pressure` at t=0
equals `water_potential` at t=0, which is itself solved inside the
hydraulic SCC.
**Status: settled.**

### Scoping for `initial` and `temporal` blocks
Both blocks are legal **inside type bodies**, alongside `relation` and
`constraint`. Declarations in a type body apply to every instance of that
type — if `XylemSegment` declares a cavitation `temporal`, then every
`XylemSegment` (whether `pathway.stem`, `pathway.leaf`, or `pathway.root[j]`
for any j) automatically evolves that way. This is the canonical form.

**Why this matters:** the "model is the science" principle (soul.md #1) says
behavior intrinsic to an entity should live on that entity, not be hoisted
to a parent scope with structural-introspection glue reaching in. The mock
Sperry cavitation logic is the worked example: the ratchet is a property
of a xylem segment, so it belongs on `XylemSegment`.

**Module-scope `initial`/`temporal` forms are retired.** The v2.0 spec's
"implicit top-level type" mechanism — where module-scope temporal/relation
declarations attached to a single distinguished type — is removed. If you
need state evolution on a top-level model type, declare it inside that
type's body.

**Per-path uniqueness rule.** Each fully-qualified temporal quantity path
receives at most one `initial` equation and at most one `temporal`
declaration, counted *after* expansion across instances. Type-body
declarations expand into per-instance equations (one per instantiation of
the type); the per-path rule catches the case where a module-scope
declaration and a type-body declaration (or two type-body declarations via
nested types) both target the same resolved path. Compile error on
duplicates, with a diagnostic identifying both sources.

**No override mechanism.** Myco has no implementation inheritance. One
declaration per path, period. If you want a variant of an entity with
different evolution, make a different type (possibly generic over behavior
parameters) and compose it. Contracts describe *interface shape*, not
*state evolution* — contracts cannot declare `initial`/`temporal`, and they
have no override story for these forms. This keeps interface and
implementation cleanly separated.

**Locus-scoped temporals.** `temporal name on locus:` is legal by symmetry
with `relation name on locus:`. State evolution that applies only at a
specific locus of a domain (e.g., surface evaporation at a soil domain's
top boundary, distinct from bulk diffusion) is expressible directly:

```myco
type Soil : Domain<G = Euclidean<Dim = 2>> as (x: Scalar<m>, y: Scalar<m>) {
    field moisture: Fraction

    temporal bulk_diffusion:
        d(moisture) = diffusion_coeff * laplacian(moisture)

    temporal surface_drying on boundary where y = surface_y:
        d(moisture) = -evap_rate(wind, radiation)
}
```

The emitter restricts the locus-scoped temporal's factor to the locus,
same way it already does for locus-scoped relations.

**Status: settled.**

### `constraint name: inequality`
Enforcement-only. Inequalities, logical predicates. Not solver-eligible like
relations — these are bounds the compiler enforces.
**Status: settled.**

### `property name: continuous_invariant`
Continuous properties like monotonicity. Quantified over full admissible domain.
Compiler verifies symbolically or errors.
**Status: settled.**

### Inline constraint sugar
`water: Potential { self <= 0 MPa }` is sugar for a named constraint at
containing type scope. `self` refers to the declared field, sibling names in
scope.
**Status: settled.**

### `let` bindings
Lexical scope, named subexpressions. Not graph quantities.
**Status: settled.**

### `if/else`, `where` conditionals
Value expressions in relation bodies.
**Status: settled for compile-time predicates.** Open: `where` on runtime values
(e.g., `where size_ratio > 1.0` in the fish predation example) is piecewise
function behavior. How does this interact with differentiability? Does the
compiler need to know about discontinuities?

## Values & Literals

### `Scalar<U>` — dimensioned quantity
The built-in parameterized type for "a real number measured in unit U."
**Status: settled.**

### `universal name: Type = value`
Physics/math constant with inline default. Overridable from Python.
`universal R: Scalar<J_mol_K> = 8.314`.
**Status: settled.**

### ~~`param name: Type`~~ — REMOVED
Killed. Redundant in an acausal system — the whole point of Myco is that any
unbound quantity is automatically free. `universal` covers named constants with
defaults; everything else is just a field whose value comes from wherever the
system determines it should (data binding, solver, learning). No need for a
separate keyword to mark "this value comes from outside."
**Status: removed.**

### No bare dimensioned literals
Compiler rejects any dimensioned number not attached to a `universal`.
Dimensionless integers (0, 1, 2) and dimensionless ratios are exempt.
**Status: settled.**

### Unit system
Dimension exponent vectors, algebraic checking, base SI internal storage. Affine
units for temperature. `value_in(unit)` as escape hatch for empirical equations.
**Status: settled.**

## Compiler Primitives

### `deriv(A, B)` — symbolic partial derivative
Resolved at compile time via chain rule on the expression graph. No runtime cost.
Always symbolic — all registered functions have transparent bodies.
**Status: settled.**

### `integrate(expr, over = domain)`
Definite integral. Symbolic first, numerical quadrature fallback. Works over
domain-typed entities: `integrate(horse.metabolic_rate, over = horse)` because
the compiler knows the entity's extent and metric.
**Status: settled.** Extends naturally to 2D/3D via the domain's metric tensor.
Open: does the compiler choose quadrature scheme or does the user configure it?

### `sum(expr for x in collection)` / `product(...)`
Aggregation over collections. Works with both index-style and iterator-style.
**Status: settled.**

### `any`, `all`, `count`, `argmin`, `argmax`
New aggregation primitives. Generator expression style:
```
any_stressed = any(t.water < threshold for t in trees)
all_healthy = all(t.water > threshold for t in trees)
n_alive = count(trees)
let tallest = argmax(t.height for t in trees)
```
`argmin`/`argmax` return the element (not index). Ties broken by index order.
Differentiability: `subgradient` (same class as `max(a, b)`). Empty collection:
`sum`=0, `any`=false, `all`=true (vacuous), `min`/`max`/`argmin`/`argmax`
undefined (requires `count > 0` guard).
**Status: new, settled.**

### Builtins — `min`, `max`, `exp`, `sqrt`, `abs`, etc.
Standard mathematical functions.
**Status: settled.**

## Controllers (formerly "Slots")

### ~~`slot` keyword~~ — REMOVED
The `slot` keyword has been collapsed. What used to be a slot is now just
**a node declared in the .myco with a type but no relation**. The compiler
sees it as unowned. At workflow bind time, Python supplies an owner.

This eliminates a whole parallel concept. A regular node and a "slot node"
were doing the same structural job — occupying a position in the graph and
requiring an owner. Keeping them as one concept reduces language surface,
aligns with the soul principle "the compiler does the work," and dissolves
two Tier 1 open questions at once (wildcard-slot `[*]` and
shared-controller portability).

In the .myco:
```myco
type SperryTree<V, P, N_SOIL, N_CANOPY> : Tree {
    canopy_layers: [LeafGasExchange<P = P>; N_CANOPY]
    // canopy_layers[i].g_w has type Conductance but no relation assigning it.
    // The compiler notes it as unowned. The workflow layer must supply an owner.
}
```

In Python (workflow layer):
```python
experiment.bind_controller(
    target="tree.canopy_layers[*].g_w",
    controller=stomatal_nn,          # any callable (NN, GP, heuristic)
    input_contract=Tree,             # what the controller consumes
)
```

The controller sees exactly what the `Tree` contract enumerates — no more, no
less. Contract-as-visibility-boundary is strict: the controller cannot reach
through to internal fields beyond what the contract declares. Expanding
visibility requires expanding the contract.
**Status: settled (collapse decided).**

### Owner types at workflow bind time
A previously-unowned node can receive any of the following owners at
workflow bind time:
- `assume_constant(path, value)` — fix to a scalar
- `assume_series(path, values)` — fix to a time series
- `learn_constant(path, init)` — learnable scalar
- `learn_trajectory(path, init)` — learnable series
- `bind_controller(path, fn, input_contract)` — callable (NN, GP, etc.)
- `bind_topology(...)` — topology owner (separate mechanism per geometry design)

Exactly one owner per node. The compiler checks this at bind time.
**Status: settled.**

### The controller's visibility is its input contract
The `input_contract` argument to `bind_controller` names a data contract.
The controller receives exactly the fields that contract declares. This is
the mechanism for shared-weights-across-studies: every study satisfies the
same `Tree` contract (even with different internal structure), so the
same controller instance can be threaded through all of them during
multi-study training. The controller never sees per-study internals — only
the contract-declared interface.
**Status: settled.**

### Transparent heuristic controllers
Sperry/Potkay-style analytic heuristics remain expressible as regular `.myco`
modules. At bind time, the Python binding may target a heuristic module
instead of a learned callable — the interface is the same (the `input_contract`
argument). This preserves the "pretrain with heuristic, fine-tune with data"
workflow without requiring special syntax.
**Status: settled.**

### SCC participation
If the controller's contract-declared inputs include nodes that are part of
an SCC (e.g., leaf water potential inside the hydraulic loop), the controller's
output node joins that SCC. The controller is opaque to symbolic reasoning
(no `deriv(·)` through it), but contributes to the residual graph as a black-
box factor. Gradient flow at train time goes through the controller's native
autodiff (JAX/PyTorch). SCC structure is fully determined only after workflow
binding, which is fine — the planner already runs post-binding.
**Status: settled.**

## Initialization & Workflow (Python side)

### `assume_constant`, `assume_series`
Fix quantity values from data. Part of the four-way workflow vocabulary:
`assume` (provide data), `observe` (provide evidence), `learn` (make
trainable), `bind` (provide controller or topology). Renaming `assume_*`
to `bind_*` was proposed and reverted — it collapsed two distinct concepts
(data assumption vs controller binding) into one verb. `assume` in the
mathematical sense is precise.
**Status: settled.**

### `bind_controller(path, fn, input_contract)`
Attach a callable (neural net, GP, ensemble, analytic heuristic module) as
the owner of an otherwise-unowned node. The `input_contract` names a data
contract — the controller's visibility is exactly what that contract
declares. Enables shared controllers across multi-study training: every
study's concrete model satisfies the same contract, so the same controller
instance threads through all of them.
**Status: settled.**

### `bind_topology(...)`
Standalone topology binding — structurally different from data assumption
(geometry/connectivity, not scalar values). See Geometry section.
**Status: settled.**

### `learn_constant`, `learn_initial`, `learn_trajectory`
Learned values. `learn_trajectory` is the fourth mutually exclusive
initialization mechanism, owns all timesteps including t=0.
**Status: settled.**

### `dt` as a normal quantity
Not magic. Can be constant or per-step.
**Status: settled.**

### Multi-binding compilation
Same model, N binding sets, shared controller. Essential for sparse multi-study
training.
**Status: settled.**

## Compiler Analysis

### SCC detection
Automatic algebraic loop discovery. No solve blocks.
**Status: settled.**

### Hierarchical SCC decomposition
When `deriv(A, g_s)` feeds back into its own SCC: inner physics Newton + IFT
boundary + outer root-find over control variables.
**Status: settled.**

### Residual graph
Core semantic object. Factor graph with variable nodes, derived nodes, residual
factors, slot nodes.
**Status: settled.**

### Four-way component classification
Computational redundancy, square implicit, underdetermined residual,
overconstrained residual.
**Status: settled.**

### Three-way overdetermination classification
The compiler distinguishes three cases when a system has more equations than
unknowns:

1. **Overdetermined but consistent (redundant).** Equations agree — one is
   derivable from the others. Example: `y = 2x`, `z = 3x`, `y + z = 5x`.
   Compiler warns, drops the redundancy. Closure policies apply here if the
   user wants to choose which to keep.

2. **Provably inconsistent.** Symbolic elimination produces a contradiction
   like `0 = 5` or `1 = 2`. No solution exists regardless of data. **Hard
   compile error** with a clear explanation ("relations on lines 47 and 93
   imply 1 = 2"). Not a closure policy situation — this is a bug in the model.

3. **Conditionally inconsistent.** Consistency depends on parameter values not
   known at compile time. Compiler emits a **runtime assertion** that fires
   before the solver runs if the concrete values produce a contradiction.

This classification also applies to `convert ... where predicate` and
`rule ... where predicate`:
- Predicate provably always true → elide the check
- Predicate provably always false → compile error
- Predicate undecidable at compile time → runtime assertion

**Status: new, settled.**

### Closure policies
For overdetermined-but-consistent systems (case 1 above). Explicit
approximations that relax the system.

**Semantic interface.** Spec §14.6 is authoritative: a closure policy
receives the target quantity and candidate paths (N = M + 1 common case:
two candidates; general case: all C(N, M) maximal square subsystems).
Candidates are named by the producing relation (e.g., `"demand_transpiration"`)
and ordered lexicographically for cross-version stability. Policies are
ordinary `.myco` functions whose arguments are candidate values and user
hyperparameters supplied at `closure_config` time. Custom policies only
operate on values — they do not access compiler-internal metadata.

**v2.1 stdlib — three policies only.**
- `weighted_average` — arithmetic mean of candidate values.
- `soft_select(preference, sharpness)` — differentiable soft selection with
  a preference ranking of relation names.
- `hard_select(preference)` — pick one path, discard the rest.
  Non-differentiable; rejected in `train` mode unless discarded paths have
  no learned parameters upstream.

All three are ordinary `.myco` functions users could write themselves.

**`condition_weighted` deferred.** Weighting paths by numerical conditioning
requires a `condition_of(expr)` primitive that inspects expression
structure (parallel to `deriv`), or the policy must be a compiler-provided
black box. Both options have real cost, and most v2.1 use cases reconcile
overdetermined systems via controller + consistency loss rather than via
conditioning-aware closure. Ship `condition_weighted` post-v2.1 if demand
emerges, at which point pick intrinsic vs black-box with evidence.

**Strike structural-introspection language.** Spec §8.5 (line ~2073) says
operation-algebra metadata is available to policies "via structural
introspection on the competing paths." Structural introspection was killed
in v2.1 (see "Iteration — Compile-Time Structural Introspection" above).
In v2.1, custom policies receive only values and hyperparameters; any
metadata-aware policy is a compiler intrinsic (deferred).

**Status: settled.**

### Plan inspection
Compilation plan as inspectable artifact. Shows SCCs, solver strategies,
symbolic resolutions, numerical fallbacks.
**Status: settled.**

### `#[verified_externally]`
Escape hatch for properties the compiler can't prove.
**Status: settled.**

### Training emission and constraint enforcement
What the compiler emits to make training stable, and where the division
of labor sits between compiler and workflow. Adopted position: compiler
does plumbing, workflow makes science choices about training dynamics.
Myco is not an ML framework — it exposes rich metadata and emits
backend-agnostic code, and the training recipes are Python library code.

**Warm-started solvers — compiler-emitted, default on.** Iterative solvers
(Newton, fixed-point) emitted by the compiler accept the previous tick's
solution as their initial guess. When state is continuous, warm-start is
almost always a win; when it isn't (cold-start evaluation, regime
changes), the workflow can disable it. This is a compilation detail, not
a science choice — it does not change the emitted residuals or the
forward value. Backend-agnostic: every solver library takes an initial
guess as an argument.

**Refinement-type bounds surfaced to the workflow.** When an unknown has
bounds from its refinement type (`type Fraction : Scalar<ratio>
{ 0 <= self <= 1 }`, `type Conductance { self >= 0 }`, etc.), the
compiler surfaces those bounds as workflow-visible metadata on the
unknown. The compiler does **not** auto-emit projection: the choice of
projection flavor (hard clip, sigmoid reparameterization, soft clip) is
a training-dynamics decision that varies by problem.

Stdlib workflow helpers ship projection options:
```python
training_config.projection = {
    "tree.stomatal_conductance": myco.projection.sigmoid(),
    "soil.water_content":        myco.projection.hard_clip(),
    "*":                         myco.projection.none(),
}
```
Backend-agnostic: all three flavors are elementary ops (`clamp`,
`sigmoid`, smooth transitions) available in every backend.

**Per-residual exposure for equality consistency losses.** Overdetermined
components (the overconstrained residuals from section 12.3) expose each
residual individually, named by the producing relation (per §14.6
conventions). `model.residuals` is a workflow-visible list of
`Residual` objects with names and runtime values.

Stdlib ships two loss helpers in v2.1:
- **`soft_penalty(weights)`** — default. Sums `w_i · r_i²` over residuals.
  `consistency_loss_weight` (per-residual or uniform) is surfaced
  prominently in training config. No "sensible" default — tuning per
  problem is documented as required.
- **`augmented_lagrangian(weights, mu, lambda_init, mu_schedule)`** —
  opt-in for brittle-penalty regimes. Adds a linear dual term
  `λ_i · r_i` with `λ_i` updated after each training step. Per-residual
  dual state is maintained by the helper; update rule defaults to the
  standard multiplier update. Two API shapes by paradigm:
  PyTorch-style mutable `lambdas.update(residuals)` or JAX-style pure
  `(params, lambdas, opt_state) = update(params, lambdas, opt_state, residuals)`.

Both helpers read the same `model.residuals` list and produce a scalar
loss term for the optimizer.

**Deferred beyond v2.1.** Homotopy continuation (blending controller
output with a baseline via annealing `α`) and pre-training against a
hand-coded heuristic are entirely workflow-layer patterns that the
controller-binding API already supports — no language features needed.
They are deferred as documented recipes, not as future spec additions.

**What is not auto-emitted by the compiler.** Projection flavor choice,
loss aggregation, dual-variable updates, annealing schedules. These are
training-dynamics choices that belong in the workflow.

**Status: settled.**

## NEW: Geometry & Domains

Full design in `planning/v2/v2.1_chunk_reports/01_geometry_design_report.md`. Summary of settled
decisions below.

### `geometry` — compile-time structural descriptor
A geometry is a first-class language construct declared with the `geometry`
keyword. Describes the *kind* of space (metric structure, topology class, named
singularities) but not any specific space. Geometries are reusable — the same
`Euclidean<Dim = 2>` serves pastures (meters) and image domains (pixels).

A geometry declares:
- **`chart`** — geometry-private coordinate binders for readable metric
  expressions. No units. Not visible outside the geometry. Lowered to positional
  slots in IR. The domain type's `as` clause provides public names and units.
- **`topology`** — connectivity class: `manifold<Dim>` for smooth regions,
  `rooted_tree` for branching networks (anatomical parent/child, NOT flow
  direction), `metric_graph` for undirected networks. Extensible.
- **`metric`** — covariant metric tensor. The fundamental primitive from which
  the compiler derives `grad`, `diverg`, `laplacian`, `curl`. For manifolds: DxD
  matrix. For graphs: per-edge `[[1]]`.
- **`locus`** — named structural singularities with structural predicates (see
  below).
- **`requires`** — geometry coefficients (scalar, spatially constant,
  time-invariant). NOT generics — per-instance runtime values resolved at
  planning time. Bound via parentheses: `Sphere(radius = my_radius)`.
- **`identify`** — periodic seam declarations. v2.1: scalar fields only.
  Vector/tensor seam transforms deferred.

```
pub geometry Euclidean<Dim: val> {
    topology manifold<Dim = Dim>
    metric = identity(Dim)
}

pub geometry BranchingManifold {
    topology rooted_tree
    metric on edge = [[1]]
    locus junction where degree >= 3
    locus terminal where degree = 1
}

pub geometry Sphere {
    chart (theta, phi)
    requires radius: Scalar<m>
    topology manifold<Dim = 2>
    metric = [[radius^2, 0], [0, radius^2 * sin(theta)^2]]
    locus pole where theta in {0, pi}
    identify phi = 0 <-> phi = 2 * pi
}
```

**What a geometry does NOT contain:** coordinate names/units, embedding in
ambient space, runtime topology data, discretization choices, time-varying or
field-dependent quantities.

**Explicit non-goal:** Time-varying geometries (moving meshes, ALE methods,
metrics that depend on fields or evolve in time). Geometry coefficients are
constrained to be scalar, spatially constant, and time-invariant — compiler
enforces as hard error.
**Status: new, settled.**

### `Domain<G>` — type-level annotation
An ordinary composite type annotated with `Domain<G = SomeGeometry>`. Tells the
compiler that instances are regions of space with continuous fields. Provides
coordinate names/units via `as`, extents via `in`, geometry coefficient bindings
via parentheses, fields, relations, locus-scoped relations.
```
type HydraulicTree : Domain<G = BranchingManifold>
    as (s: Scalar<m>)
{ ... }

type FruitSurface : Domain<G = Sphere(radius = radius)>
    as (lat: Scalar<rad>, lon: Scalar<rad>)
{
    radius: Scalar<m>    // regular field, feeds geometry coefficient
}
```
**Status: new, settled.**

### Coordinate naming — `as (name: Type, ...)`
Declared on the domain type definition. Coordinates carry units. Number of names
must match geometry's chart dimensionality (positional mapping). Shared by all
fields over that domain.
**Status: new, settled.**

### Domain extents are regular variables
Domain bounds are not const generics — they're regular variables that can be
set from data or even change during simulation. Declared inline with the
coordinate naming:
```
type Horse : Domain<G = Interval>
    as (b: Scalar<m> in 0 m .. body_length)
{ ... }
```
The `in low .. high` syntax specifies the extent. `body_length` is a regular
field on the type — per-instance, can grow over time.
**Status: new, settled.**

### `locus` — named structural singularity
Declared inside a `geometry` block with a structural predicate. Named classes
of topologically special points (junctions, terminals, poles).

Two purposes:
1. **Default condition generation:** Compiler generates flux balance at graph
   junctions when triggered by `diverg()` usage.
2. **User condition targets:** User writes locus-scoped relations for continuity,
   boundary conditions, overrides.

Locus `where` predicates reference both built-in structural facts (`degree`,
`depth`, `strahler_order`) and workflow-supplied topology metadata tags (`role`):
```
on junction where degree = 3
on terminal where role = "leaf"
on junction where strahler_order >= 3
```
**Status: new, settled.**

### `trace()` — directional limit at graph vertices
`trace(field, edge = label)` evaluates a field's limiting value as you approach
a locus from a specific direction. Specific to graph topologies.

Edge labels for `rooted_tree`: `parent`, `children[i]` (anatomical labels, NOT
flow direction). Both `rooted_tree` and `metric_graph` expose `incident_edges`
with `orientation(e)` for general iteration.

```
// Pit membrane drop — parent/children sugar:
relation pit_drop on junction:
    for c in children:                                       // (*)
        trace(pressure, edge = parent) - trace(pressure, edge = c)
            = pit_resistance * trace(axial_flux, edge = c)

// General incident-edge form:
relation flux_balance on junction:
    sum(e in incident_edges,
        orientation(e) * trace(axial_flux, edge = e)) = 0
```

**(\*) Note:** `for c in children` and `sum(e in incident_edges, ...)` are
graph-neighborhood iteration — intended semantics shown here, exact syntax
pending general dynamic iteration design.

At terminals (degree 1), `edge` argument is optional. Using `trace` outside a
locus-scoped relation is a compile error.
**Status: new, settled.**

### Locus-scoped relations — `relation name on locus_name:`
Relations scoped to a specific locus. Either provide a condition where no
default exists, or override a compiler-generated default.

**Default generation rule:** When the compiler sees `diverg(flux_field)` in any
conservation-form equation, it generates `balance(flux_field)` (Kirchhoff
flux balance) at unhandled graph junctions. **No continuity auto-generated.**
Users opt in explicitly:
```
// Explicit continuity (stdlib convenience function):
relation pressure_continuous on junction:
    continuous(pressure)

// Or the Kirchhoff bundle:
relation standard_junction on junction:
    kirchhoff(pressure, axial_flux)
```

**`replaces` with obligation keys** for overriding generated defaults:
```
relation leaky_junction on junction replaces balance(axial_flux):
    sum(e in incident_edges,
        orientation(e) * trace(axial_flux, edge = e)) = leak_rate
```
Obligation keys (`balance(axial_flux)`) are stable semantic identifiers, not
user-chosen relation names. Plan inspection shows which default was replaced.

**Terminal coverage** is a planner-stage global well-posedness check, not a
per-terminal syntactic rule. Terminals always require explicit user conditions —
no auto-generation.

`where` predicates on locus-scoped relations discriminate by metadata:
```
relation leaf_outflow on terminal where role = "leaf":
    trace(axial_flux) = transpiration_sink

relation root_boundary on terminal where role = "root":
    trace(pressure) = soil_water_potential
```
**Status: new, settled.**

### Manifold boundary conditions
For manifold geometries, boundaries use the `boundary` keyword (not `trace`).

**1D (settled):**
```
boundary b = 0 m:
    body_temp = nose_temp                                    // Dirichlet

boundary b = body_length:
    -k * normal_grad(body_temp) = h * (body_temp - ambient)  // Robin
```

`normal_grad(field)` is the intrinsic outward co-normal derivative at a
boundary. On curved surfaces, this is the co-normal within the manifold, NOT the
ambient 3D surface normal. Unifies Neumann/Robin conditions across dimensions.
In 1D: spatial derivative with endpoint-determined sign. In 2D/3D:
`grad(field) . outward_co_normal`.
**Status: new, 1D settled.** Open: 2D/3D boundary selectors beyond
`coord = value`, non-axis-aligned boundaries, internal interfaces.

### Embedding is separate from intrinsic geometry
A geometry describes intrinsic structure only. How a domain is embedded in
ambient 3D space lives on the entity as regular scalar fields:
```
type Tree {
    x: Scalar<m>      // position in forest
    y: Scalar<m>
    hydraulics: HydraulicTree   // contains the 1D vascular network
}
```
Cross-domain coupling at the embedding (e.g., 1D root network meets 3D soil) is
handled by kernel coupling, not the geometry system.
**Status: new, settled.**

### `bind_topology` — workflow-layer topology injection
For geometries with data-driven topology (branching networks, irregular meshes),
the Python workflow layer provides the specific graph/mesh structure:
```python
model = myco.load("forest.myco")
tree_data = load_xylem_network("tree_42.graphml")
model.bind_topology("tree_42.hydraulics", tree_data)
```

`bind_topology` validates at schema level only: structural correctness, data
completeness, unit consistency. Well-posedness (enough boundary conditions,
correct constraint counts) is a planner-stage check.

**Workflow naming:** `bind_topology` is a standalone new API, not part of a
broader rename. `assume_constant` and `assume_series` remain unchanged —
see Initialization & Workflow section. `bind_topology` uses `bind` because
topology injection is structurally different from assuming a quantity's value
(it provides graph structure, not a scalar or series).

Each topology class defines its data schema. For `rooted_tree`: vertices, edges,
root, edge_lengths (units must match `as` clause), vertex_tags (must cover all
`where`-referenced keys). For `metric_graph`: same without root. For manifolds:
no `bind_topology` needed — mesh resolution configured via
`experiment.compile(spatial_config=...)`.
**Status: new, settled.**

### Standard library geometries
Shipped as stdlib, not compiler built-ins. Users define custom geometries with
the same primitives.

| Name | Topology | Metric | Loci | Typical use |
|---|---|---|---|---|
| `Interval` | `manifold<1>` | `[[1]]` | none | Soil columns, body axes |
| `Euclidean<Dim>` | `manifold<Dim>` | `identity(Dim)` | none | Pastures, canopy layers |
| `Polar` | `manifold<2>` | `[[1,0],[0,r^2]]` | `pole` | Radial symmetry |
| `Sphere` | `manifold<2>` | curved (requires `radius`) | `pole`, `identify` | Fruit, globes |
| `BranchingManifold` | `rooted_tree` | `[[1]]`/edge | `junction`, `terminal` | Xylem, phloem |
| `MetricGraph` | `metric_graph` | `[[1]]`/edge | `junction`, `terminal` | Pipe networks |

Future: `rooted_graph` for cyclic-but-rooted structures (anastomosing roots,
leaf vein networks, fungal mycelium). Architecture should not prevent it.
**Status: new, settled.**

### Discretization
Compiler owns spatial discretization. User declares continuous equations;
compiler chooses a discrete approximation. Override is Python-side config:
```python
experiment.compile(
    spatial_config={
        "HydraulicTree": {"method": "finite_volume", "resolution": 0.01},
        "Pasture": {"method": "finite_difference", "nx": 50, "ny": 50}
    }
)
```
Consistent with the principle: `.myco` describes the world, workflow layer
controls how to solve it.
**Status: new, settled.**

## NEW: Fields

### `field name: Type` — implicit over containing domain
Inside a domain type, `field temp: Temperature` is implicitly over that domain.
The `over` keyword is only needed for subdimensional fields — fields that vary
over fewer dimensions than the containing domain:
```
// In a 2D domain with as (px: Scalar<m>, py: Scalar<m>):
field nitrogen: Scalar<g_m2> over py    // 1D field varying only along py
```
The `over` coordinate must reference a name from the domain type's `as` clause.
Scalars are degenerate fields (a field sampled at one point). The compiler
understands this internally.

**Graph field semantics:** On graph topologies, fields live on **edge
interiors**. Vertices are loci where traces are taken. This distinction matters
for `grad` (defined on edge interiors), `trace` (defined at vertices), and
finite-volume lowering.

**Vertex-local 0D state** (storage, capacitance, reaction terms at graph
vertices) is declared using `field name on locus_name: Type`:
```
field junction_volume on junction: Scalar<m3>
field junction_pressure on junction: WaterPotential
```
Compiler allocates locus-scoped fields as flat arrays indexed by locus instances
(not mesh arrays). Locus-scoped fields compose naturally with `trace()`.

**Status: new, settled.** Fields lower to arrays on a mesh after discretization.
Spatial operators derive from the domain's metric.

### Field sampling — `.at(x, y)`
Read a field's value at specific coordinates. Used for coupling discrete
entities to fields.
**Status: new, concept settled.** Open questions:
- Is `.at()` the right syntax?
- What about interpolation between mesh points after discretization?
- Does sampling a field at a point return an exact value or an interpolated
  approximation? (Probably the latter, but should the user know?)

### Spatial operators — `grad()`, `diverg()`, `laplacian()`, `curl()`, `normal_grad()`
Derived from the domain's metric tensor. `grad(scalar_field) -> vector_field`,
`diverg(vector_field) -> scalar_field`, etc. `diverg` (not `div`) to avoid
confusion with division — final name settled.

`normal_grad(field)` is the intrinsic outward co-normal derivative at manifold
boundaries. Unifies Neumann/Robin conditions across 1D, 2D, 3D.

On graph topologies, `grad` is defined on edge interiors (from the per-edge
metric). At junctions, `trace()` provides the directional limit mechanism — see
Geometry & Domains section. `diverg()` in a conservation equation triggers
automatic `balance()` generation at junctions.

Unit system extends through domains: spatial operators derive dimensional
signatures from coordinate units. `grad` divides by coordinate dimension,
`integrate` multiplies. The compiler infers this from the coordinate type
declarations. No new user-facing syntax needed.
**Status: new, settled.** Open: full catalogue of operators and dimensional
signatures needed. Subdomain operator interaction undesigned.

## NEW: Dynamic Topology

### `event name: InputTypes -> OutputTypes { field mapping }`
Discrete topology changes. Renamed from `rule` — "rule" is too generic and
overlaps conversationally with `relation` (mathematical rules). "Event" clearly
communicates discrete, conditional, topology-changing behavior. All topology
changes are discrete, so all topology changes are events. Non-event discrete
behavior (mode switching) is handled by conditional relations. `rule` is killed
— no remaining use case.

Inputs are consumed from the graph, outputs are produced. Drop is implicit —
if two go in and one comes out, one was consumed.

Events declare WHAT is valid (the topology transition and field mapping). The
`when` clause declares the condition. No match/select/schedule logic in the
language — selection emerges from the continuous dynamics or is configured
from the workflow layer.

```
event predation: Fish, Fish -> Fish
    when predator.mass > 2.0 * prey.mass
{
    survivor.mass = predator.mass + prey.mass
}

event reproduction: Fish -> Fish, Fish
    when parent.reproductive_investment > threshold
{
    parent_after.mass = parent.mass - offspring_cost
    baby.mass = offspring_cost
}

event bird_arrival: -> Bird
    when count(pond.fish) > 100
{
    bird.mass = 1.0 kg
}
```

**Status: new, settled.** Renamed from `rule` to `event` — `rule` is too
generic and overlaps conversationally with `relation`. `event` clearly
communicates "discrete thing that happens."

**Container scoping rule.** An event is declared on the smallest container
type whose scope contains all its input/output participant types. For the
common case (all participants live in one container's `some`-sized
collection), this is the container itself — same as relations. For
cross-container events, the event lives on the nearest common ancestor
container. When a participant type appears as a pool in multiple sibling
children, the signature uses dotted paths (`pond.fish -> sky.birds`);
otherwise the bare type is unambiguous.

```
type Ecosystem {
    pond: Pond
    sky: Sky

    event migration: Fish -> Bird
        when season == winter and pond.fish_density > threshold
    {
        bird.mass = fish.mass * conversion_efficiency
    }
}
```

The event body may reference quantities on all participant types and on
the container it lives on, as with single-container events.

**Heterogeneous dynamic collections:** Events targeting `[T<impl Contract>; some]`
must specify the concrete output type: `event oak_recruit: -> Tree<FarquharC3>`.
This lets the compiler route to the correct type pool.

**Generic events — first-class.** Events may be declared over type or const
generic parameters:
```
event recruit<S: Photosynthesis>: -> Tree<S>
    when seed_rain<S> > germination_threshold<S>
{
    sapling.water_potential = soil_surface_water_potential
    sapling.v_max_25 = S.v_max_25
}

event split<N: const i32>: Cell -> [Cell; N]
    when parent.size > division_threshold
{
    ...
}

event infect<H: HostSpecies, P: Pathogen>: Tree<H> -> InfectedTree<H, P>
    when contact_density<H, P> > infection_threshold
{
    ...
}
```
The compiler monomorphizes each generic event at model expansion, emitting
one concrete event per type (or const value) in scope that satisfies the
bound. Multi-parameter generic events expand over the cartesian product of
their bounds. The bound is the sole filter — the expansion set is not
configurable from the workflow layer. Context-dependent group membership
(e.g., "invasive in Study A, native in Study B") is out of scope; users
either push such distinctions into the type hierarchy or use `when`-guards
inside a non-generic event.

Named/positional generic argument rules match the type rule: positional for
a single parameter (`recruit<Loblolly>`), named required for two or more
(`infect<H = Loblolly, P = Rust>`).

**Workflow-layer addressing.** Generic events expose both the group name
and the per-instantiation name. `model.set_event_order(["recruit",
"mortality"])` addresses the whole monomorphized family (tiebreak by type
declaration order). `model.set_event_order(["recruit<Loblolly>",
"recruit<Oak>", ...])` addresses specific instantiations for fine control.
Policies (`"shuffle_types"`, custom callables) receive the expanded event
list; the group name is a convenience, not a runtime object.

Open questions:
- Can events span multiple container types?

### `when` — event trigger condition
Condition under which an event fires. Deterministic thresholds. Can reference
quantities on the participants and on the container. Edge-triggered (fires at
the moment the condition becomes true).
**Status: new, settled.**

### Event execution model
Events fire **sequentially by type**, in a deterministic order. Within a single
event type, all valid firings execute **in parallel** (GPU-batched). Consumed
entities are marked unavailable before the next event type fires.

**Firing order** is a simulation parameter, not language syntax. Default:
declaration order in the `.myco` file. User overrides from Python via
scheduling policies:
```python
model = myco.load("ecosystem.myco")
model.get_events()  # enumerate all events for inspection

# Fixed order (default: declaration order)
model.set_event_order(["predation", "reproduction", "bird_arrival"])

# Shuffle event type order each timestep (seeded for determinism)
model.set_event_policy("shuffle_types", seed=42)

# Custom callable — the slot for event scheduling. Can be a neural
# network, a handwritten heuristic, a random shuffle, whatever.
model.set_event_policy(my_scheduler_function)
```

The custom callable receives pending events and state, returns batches
to fire. Events within a batch fire in parallel (GPU-batched); batches
fire sequentially. This IS the slot for event scheduling — it lives in
Python, not in the `.myco` file, because scheduling is a workflow concern.

**Within-event conflicts** (two predators want the same prey): deterministic
tiebreak by index order, overridable from Python. The compiler can detect that
two events could compete for entities of the same type and warn.

**Conservation is order-independent.** Each event conserves individually
(compiler checks statically). The execution model guarantees no entity is
consumed twice (runtime invariant via mask updates). Total conservation holds
regardless of firing order. The ordering only affects WHERE mass ends up.
**Status: new, settled.**

### `{ conserved }` — field annotation
Annotated directly on the field declaration via conservation group hierarchy:
```
type Mass : Scalar<kg> { conserved }
type FishMass : Mass
type DetritusMass : Mass
```
Compiler checks that every event consuming/producing instances accounts for
fields with a `{ conserved }` ancestor. If an event destroys a Fish and doesn't
route its mass somewhere, compile error. See "Named types and conservation
groups" above for full design.
**Status: new, settled.** Open questions:
- Scoped conservation? `{ conserved within Pond }` for quantities that can
  leave the system via declared boundary fluxes?
- How do boundary fluxes interact with conservation? (Birds flying away with
  mass — is that a boundary condition on the container?)
- Does `{ conserved }` work for fields (continuous) as well as scalars?

### Two-tier collection sizing: `val` and `some`
Collections have two size modes:
- `[Horse; N_HORSE]` where `N_HORSE: val` — size fixed at instantiation, known
  to the compiler. Enables static Jacobian sizing.
- `[Fish; some]` — size changes at runtime as events fire. Dynamic topology.

`some` signals that this collection's size is not compile-time-known.
Renamed from `dyn` to avoid collision with `impl` (heterogeneous element
types, previously also called `dyn`).

**Bind-time vs event-time dynamism:** `some` covers both "unknown until
bind, then fixed" and "changes during rollout via events." The compiler
distinguishes mechanically: if no events target a `some`-sized collection,
it's bind-static (no validity mask updates, static Jacobian after bind).
No user annotation needed — inferred from absence of events.

**`impl` + `some` compose:** `[Tree<impl Photosynthesis>; some]` — heterogeneous
dynamic collection. Frontend desugars to per-type pools (one pool per
implementation in scope at the model module level, after whole-program assembly).
Adding an implementation to a library doesn't change anything until a model
module imports it. No explicit listing needed.

Lowering is backend-specific: PyTorch uses packed arrays with symbolic shapes;
JAX uses `[Type; MAX_CAPACITY]` with validity mask (`MAX_CAPACITY` from workflow
layer). See "Dynamic topology lowering strategy" below.

Iteration over `some`-sized collections uses `for x in collection` syntax — see
Relations section.

Full design in `planning/v2/v2.1_chunk_reports/02_collections_iteration_report.md`.
**Status: new, settled (renamed from `dyn`).**

## NEW: Coupling & Kernels

### Kernel coupling — `K(a, b) -> strength`
Soft spatial coupling between two domains or between a domain and discrete
entities. At every point in domain A, coupling to domain B is weighted by the
kernel function. Replaces crisp wiring with graded, distance-weighted
interaction. Universal interface for root-soil, trunk-canopy, shading,
competition, dispersal.
**Status: new, concept settled.** Open questions:
- Is a kernel just a function used inside an `integrate` call, or does it need
  its own declaration? (See coupling-as-sugar section below.)
- Can kernels be learned (neural slots)? Concept says yes, syntax undesigned.
- How does kernel sparsity (characteristic length scale) get communicated to
  the compiler for spatial indexing optimization?

### Coupling may be sugar for relation with kernel integration
A coupling kernel might not need its own keyword. It could be syntactic sugar
for a relation that involves integration weighted by a kernel function:
```
relation root_uptake for i in 0..N_ROOT:
    roots[i].water_supply = integrate(
        K_root(roots[i], soil) * soil.water_potential,
        over = soil
    )
```
The compiler recognizes kernel-weighted integrals as a pattern and applies
spatial optimizations (characteristic length scale → skip distant interactions).
**Status: new, open.** Needs more design work to determine whether coupling
deserves its own keyword or is just a pattern the compiler detects.

### `integrate(field, over = entity)` for domain types
When a type is a domain, you can integrate a field over it:
```
total_heat = integrate(horses[i].metabolic_rate, over = horses[i])
```
This works because `horses[i]` is a domain type with known extent and metric.
The compiler handles the quadrature. This is how emergent properties (total
mass, total heat output, etc.) are computed from fields.
**Status: new, settled.**

## NEW: Cross-Scale Visibility

### Downward-only visibility
Parents see children, children never see up. A Pasture can reference
`horses[i].body_temp`, but a Horse cannot reference `pasture.air_temp` directly.
Information flows down through explicit relations on the parent:
```
relation local_climate for i in 0..N_HORSE:
    horses[i].ambient_temp = air_temp.at(horses[i].x, horses[i].y)
```
This is composition over inheritance. Children are self-contained components
that receive their environment as input — they don't know who owns them.
**Status: new, settled.**

### Emergent properties are downward aggregations
Emergent properties (whole-organism metabolic rate, total canopy LAI, etc.) are
computed by the parent integrating or summing over its children:
```
total_metabolic = integrate(horses[i].metabolic_rate, over = horses[i])
pasture_lai = sum(i in 0..N_HORSE, horses[i].lai)
```
No upward references needed. The parent is the one who cares about the
aggregate, so the parent computes it.
**Status: new, settled.**

### No `physical` keyword
Entities in a spatial domain don't need a special `physical` marker. They just
have scalar coordinate fields:
```
type Fly {
    pos: Scalar<m>        // position along the horse's body axis
    skin_temp: Temperature
}
```
The compiler doesn't need to know an entity is "physical" — it just sees fields
and relations. Spatial coupling happens through explicit `.at()` sampling and
relations, not through a type-level annotation.
**Status: new, settled.**

## NEW: Solver Architecture (implementation, not language)

### Block-decomposed solver
Intra-entity SCCs vmapped on GPU. Inter-entity coupling through shared resources
is sparse and block-decomposable. Block Gauss-Seidel outer iteration.
**Status: concept settled.** This is an implementation detail, not language
syntax. Included here because language design choices (fields, kernels, dynamic
topology) constrain the solver architecture.

### Static/dynamic classification
Compiler mechanically classifies model modules. If the type graph contains
`some`-sized collections or events: dynamic topology (solver factory). Otherwise:
fully static (fixed factor graph, everything proven at compile time). Superset,
not replacement.
**Status: new, settled.**

### Backend targets
Primary target: **PyTorch** (`torch.compile(dynamic=True)` supports symbolic-
shape kernels that don't recompile when dimensions change — natural fit for
dynamic topology). Secondary: **JAX** for static-topology models where XLA
optimization shines. Long-term watch: **Enzyme + Rust** (LLVM-level AD of
native Rust/CUDA code, not stable until ~2027).

Myco is a compiler — the backend is a lowering target, not a language decision.
`[Fish; some]` means the same thing regardless of backend.

### Dynamic topology lowering strategy
`[Fish; some]` compiles to a **fixed-size array with validity mask**. The
workflow layer must supply `MAX_CAPACITY` when binding the experiment. The
`.myco` file says "this can grow/shrink" (`dyn`), the experiment runner says
"but never more than 500." Rule firing flips mask bits and overwrites slot
contents — no array resizing, no recompilation.

On PyTorch: `torch.compile(dynamic=True)` can handle actual resizing without
recompilation, so masking may be optional. On JAX: masking is mandatory (XLA
requires static shapes). The compiler emits the appropriate lowering per backend.

**Status: new, settled.** Based on external review (Gemini flagged JAX
limitation) and backend research (April 2026).

### Temporal equation lowering
Two surface forms with distinct semantics:

- **`d(x) = expr`** — continuous ODE. The user declares the instantaneous
  rate; the compiler chooses the integration scheme (Euler, RK4, implicit,
  adaptive) and internal step size. For stiff systems (diffusion, fast
  chemistry), implicit. For non-stiff ODEs, explicit is fine. The compiler
  owns both spatial discretization (mesh) and temporal discretization (Method
  of Lines).
- **`step(x) = expr`** — discrete update. The compiler applies the RHS
  verbatim at each tick. RHS reads prior-tick values; LHS writes current-tick
  values. Used for accumulators, ratchets, counters — quantities with no
  continuous derivative (e.g., `min_historical_pressure`).

The old `[t+1] = [t] + dt * rate` form is gone: it hardcoded Forward Euler
(unconditionally unstable for diffusion Laplacians) and violated soul
principle #3 by embedding numerics in the model.

**Status: settled.** Spec §6.3 and mock_sperry updated.

### Performance targets
~3,000-3,400 tree-years/second on H100 for Sperry-scale models. Hectare-century
in ~17s forward, ~3min training. Memory-bound around ~10M entities.
**Status: back-of-envelope estimates.** Need validation once implementation
exists.

## Worked Example: Fly on a Horse in a Pasture

This example exercises domains, fields, spatial operators, cross-scale coupling,
subdimensional fields, boundary conditions, and discrete entities with scalar
positions. Three nested scales: pasture (2D) → horse (1D body axis) → fly
(scalar).

```myco
type Pasture<N_HORSE: val> : Domain<G = Euclidean<Dim = 2>>
    as (px: Scalar<m>, py: Scalar<m>)
{
    field air_temp: Temperature
    field wind_speed: Scalar<m_per_s>

    horses: [Horse; N_HORSE]

    relation local_climate for i in 0..N_HORSE:
        horses[i].ambient_temp = air_temp.at(horses[i].x, horses[i].y)
        horses[i].ambient_wind = wind_speed.at(horses[i].x, horses[i].y)
}

type Horse<N_FLY: val> : Domain<G = Interval>
    as (b: Scalar<m> in 0 m .. body_length)
{
    // position in parent domain
    x: Scalar<m>
    y: Scalar<m>

    // physical parameters
    body_length: Scalar<m>
    k: Scalar<W_per_m_K>
    h: Scalar<W_per_m2_K>

    // environment (pushed down by parent)
    ambient_temp: Temperature
    ambient_wind: Scalar<m_per_s>

    // fields over the body axis
    field body_temp: Temperature
    field metabolic_rate: Scalar<W_per_m3>

    // heat equation along the body
    temporal thermoregulation:
        d(body_temp) = k * laplacian(body_temp)
            + metabolic_rate
            - h * (body_temp - ambient_temp)

    // boundary conditions at body endpoints
    boundary b = 0 m:
        body_temp = nose_temp
    boundary b = body_length:
        -k * normal_grad(body_temp) = h * (body_temp - ambient_temp)

    // discrete entities living on this domain
    flies: [Fly; N_FLY]

    relation fly_environment for i in 0..N_FLY:
        flies[i].skin_temp = body_temp.at(flies[i].pos)
}

type Fly {
    pos: Scalar<m>              // position along horse body axis
    skin_temp: Temperature      // pushed down by parent
    universal comfort_midpoint: Temperature = 310 K
    universal comfort_steepness: Scalar<1_per_K> = 0.5

    relation comfort:
        happiness = 1.0 / (1.0 + exp(comfort_steepness * (skin_temp - comfort_midpoint)))
}
```

**What this demonstrates:**
- **`Domain<G>`**: Pasture is `Domain<G = Euclidean<Dim = 2>>`, Horse is
  `Domain<G = Interval>`. Fly is not a domain — just a discrete entity.
- **Coordinate naming with units**: `as (px: Scalar<m>, py: Scalar<m>)`.
- **Domain extents as variables**: `in 0 m .. body_length` — not const generics.
- **Implicit field over**: `field body_temp: Temperature` is implicitly over the
  horse's body axis.
- **Cross-scale coupling via `.at()`**: `air_temp.at(horses[i].x, horses[i].y)`
  samples the pasture field at a horse's position.
- **Downward-only visibility**: Pasture pushes climate to horses, horses push
  skin temp to flies. No upward references.
- **Scalar coordinates, no `physical` keyword**: Fly has `pos: Scalar<m>`, not
  `physical position: Point over parent`.
- **Value parameters**: `N_HORSE: val`, `N_FLY: val`.
- **Boundary conditions with `normal_grad`**: `normal_grad(body_temp)` is the
  intrinsic co-normal derivative, correct for Robin conditions in any dimension.
- **Spatial operators**: `laplacian(body_temp)`, `normal_grad(body_temp)` —
  derived from the `Interval` geometry's metric.

## Deferred / Out of Scope

### PDE solvers
Field-first design means the type system and spatial operators are designed now.
Actual PDE solver backends (multigrid, spectral elements, etc.) are deferred.
One simple solver (finite differences on regular grid) to test the interface.

### Recursive types
`type Branch { children: [Branch; ?] }`. Deferred — dynamic topology with
rules may subsume the need. Revisit if real users find the flat-array
representation painful.

### Stochastic language primitives
Per design decision: the world is deterministic, stochasticity is an
observational bandwidth limit. Neural slots learn hidden deterministic rules.
Noise injection is a Python-side workflow concern. No `random()` in .myco.

### Proc macros
Deferred beyond v2.

### Full E3SM representation
Dynamical cores (atmosphere, ocean) are PDE-on-mesh problems. Myco can represent
the land surface, parameterizations, and biogeochemistry. Dynamical cores
require the deferred PDE solver extension.
