# Myco v2.1 Language Features — Status & Open Questions

This document tracks every language feature, its status (settled, in progress, or
undesigned), and what remains to pin down. Features from the original v2 spec are
included for completeness; new features from the April 2026 design sessions are
marked as such.

---

## Module System

### `module path::name`
Module declaration. Provides namespacing, becomes the basis for imports.
**Status: settled.**

### `use path::{items}`
Imports. Modules may re-export.
**Status: settled.**

### `pub`
Inter-module visibility. Items private by default. `pub` controls inter-module
visibility only — workflow layer can bind any path regardless.
**Status: settled.**

### Library vs model modules
Libraries define reusable components (types, contracts, functions). Model modules
instantiate with `node`. Like Rust lib vs bin.
**Status: settled in concept.** Open: add the lib/bin analogy framing to the spec
prose (tracked in riley_spec_notes.txt).

## Type System

### `type Foo : Bar` — newtype / scalar alias
Creates a named type wrapping another. `type FishMass : Scalar<kg>`. Used for
semantic type safety — `FishMass + WaterMass` is a compile error even though both
are `kg`.
**Status: settled.** Open: clarify that "atomic" means leaf of the containment
tree (holds a numerical value), not "single-field."

### `type Foo { fields }` — composite type
Composite structural definition. Contains fields, constraints, relations.
**Status: settled.**

### `node name: Type` — instantiation
Creates a concrete instance in a model module. `node tree: SperryTree<...>`.
Reserved for model modules only — library modules use `type` exclusively.
**Status: settled.**

### Named types
Same-dimension types with different names are incompatible. `FishMass` and
`WaterMass` cannot be added. Arithmetic rules defined in spec section 4.7.
**Status: settled.**

### `convert A -> B` — named-type conversion
Explicit opt-in conversion between named types at topology transitions.
`convert FishMass -> DetritusMass`. No implicit coercion — conversions only
happen where declared and invoked explicitly.
**Status: new, partially designed.** Open questions:
- Is conversion always 1:1 (same magnitude, different type)?
- Can conversions carry a ratio (e.g., assimilation efficiency)?
- Is there a `try_convert` for fallible conversions?
- Can users define custom conversion logic (a function body) or is it always
  a dimensionally-compatible reinterpretation?

## Generics

### Type parameters — `<V: VulnerabilityCurve>`
Types and functions can be generic over contracts.
**Status: settled.**

### Value parameters — `<N_SOIL: val>`
Types can be parameterized by compile-time-known positive integers. Keyword is
`val` (not `const` or `usize` — those scare scientists). `val` signals "this is
a value, not a type" without leaking implementation details.
**Status: settled (updated from `const N: usize`).**

### `where` clauses — `where N >= 2`
Bound verification on const generics.
**Status: settled.**

### Named generic arguments
At instantiation sites, all positional or all named, no mixing.
`SperryTree<V = WeibullVC, P = FarquharC3, N_SOIL = 4, N_CANOPY = 2>`.
Names come from the type definition.
**Status: new, design accepted.** Not yet in spec. Open: nothing major, just
needs spec prose and examples.

### `dyn Contract` — heterogeneous collections
`[PFT<dyn Photosynthesis>; N]` in library modules. Users specialize per-element
in model modules. Compiler monomorphizes — `dyn` is gone before flattening. No
runtime dispatch.
**Status: settled.**

## Contracts

### `contract Name { ... }`
Trait/interface with inputs and outputs. The hybrid design: non-input members
(parameters like `b`, `c`) are real graph quantities. Input members are formal
parameters defining the invocation signature.
**Status: settled.**

### `input` / `output`
Declare the invocation signature of a contract. Inputs are functional parameters,
outputs are structural obligations.
**Status: settled.**

### Contract defaults
Default relation implementations. Included if and only if the implementing type
doesn't provide its own relation for that output.
**Status: settled.**

### Contract satisfaction
Two parts: input signature (functional) and output obligation (structural).
**Status: settled.**

---

## Functions

### `fn name<generics>(params) -> ReturnType { body }`
Registered functions with typed parameters. Compiler uses these for dimensional
analysis, symbolic differentiation, and solver emission.
**Status: settled.**

### Function-level generics
`fn arrhenius<U: Unit>(value_25: Scalar<U>, ...) -> Scalar<U>`. Same
monomorphization as types.
**Status: settled.**

### Annotation blocks
`invertibility: bijective`, `differentiability: smooth`. Declared per-function.
Compiler uses these for symbolic analysis.
**Status: settled.**

## Relations & Equations

### `relation name: equation`
Acausal equality. `=` means equality, not assignment. Compiler solves in either
direction. Relations connect quantities across types.
**Status: settled.**

### `relation name for i in 0..N:`
Iteration over fixed-size arrays. Generates one equation per element.
**Status: settled for fixed-size arrays.** Open: how does iteration work over
`dyn`-sized collections? Same `for i in 0..N` syntax? `for x in collection`?

### `temporal name: d(x) = expr`
Time derivative. Declares how a quantity evolves over time. `dt` is a normal
quantity, not a magic name.
**Status: settled.**

### `constraint name: inequality`
Enforcement-only. Inequalities, logical predicates. Not solver-eligible like
relations — these are bounds the compiler enforces.
**Status: settled.**

### `property name: continuous_invariant`
Continuous properties like monotonicity. Quantified over full admissible domain.
Compiler verifies symbolically or errors.
**Status: settled.**

### Inline constraint sugar
`water: Potential { self <= 0 MPa }` is sugar for a named constraint at
containing type scope. `self` refers to the declared field, sibling names in
scope.
**Status: settled.**

### `let` bindings
Lexical scope, named subexpressions. Not graph quantities.
**Status: settled.**

### `if/else`, `where` conditionals
Value expressions in relation bodies.
**Status: settled for compile-time predicates.** Open: `where` on runtime values
(e.g., `where size_ratio > 1.0` in the fish predation example) is piecewise
function behavior. How does this interact with differentiability? Does the
compiler need to know about discontinuities?

## Values & Literals

### `Scalar<U>` — dimensioned quantity
The built-in parameterized type for "a real number measured in unit U."
**Status: settled.**

### `universal name: Type = value`
Physics/math constant with inline default. Overridable from Python.
`universal R: Scalar<J_mol_K> = 8.314`.
**Status: settled.**

### ~~`param name: Type`~~ — REMOVED
Killed. Redundant in an acausal system — the whole point of Myco is that any
unbound quantity is automatically free. `universal` covers named constants with
defaults; everything else is just a field whose value comes from wherever the
system determines it should (data binding, solver, learning). No need for a
separate keyword to mark "this value comes from outside."
**Status: removed.**

### No bare dimensioned literals
Compiler rejects any dimensioned number not attached to a `universal`.
Dimensionless integers (0, 1, 2) and dimensionless ratios are exempt.
**Status: settled.**

### Unit system
Dimension exponent vectors, algebraic checking, base SI internal storage. Affine
units for temperature. `value_in(unit)` as escape hatch for empirical equations.
**Status: settled.**

## Compiler Primitives

### `deriv(A, B)` — symbolic partial derivative
Resolved at compile time via chain rule on the expression graph. No runtime cost.
Always symbolic — all registered functions have transparent bodies.
**Status: settled.**

### `integrate(expr, over = domain)`
Definite integral. Symbolic first, numerical quadrature fallback. Works over
domain-typed entities: `integrate(horse.metabolic_rate, over = horse)` because
the compiler knows the entity's extent and metric.
**Status: settled.** Extends naturally to 2D/3D via the domain's metric tensor.
Open: does the compiler choose quadrature scheme or does the user configure it?

### `sum(i in collection, expr)` / `product(...)`
Aggregation over collections.
**Status: settled.**

### Builtins — `min`, `max`, `exp`, `sqrt`, `abs`, etc.
Standard mathematical functions.
**Status: settled.**

## Slots & Controllers

### `slot name(inputs) -> outputs`
Neural controller placeholder. Trained via supervised loss on observations.
Replaces teleological heuristics.
**Status: settled.**

### Wildcard slots
Element-local inputs (vmapped) vs global inputs (broadcast). Partition is
structural.
**Status: settled.**

### Transparent controllers
Normal `.myco` components imported as relations. No special file format.
**Status: settled.**

### Structural identity for shared controllers
All experiments sharing a controller must use the same model instantiation.
**Status: settled.** Acknowledged as conservative — element-wise vectorized
slots may relax this in future.

## Initialization & Workflow (Python side)

### `assume_constant`, `assume_series`
Fix quantity values from data.
**Status: settled.**

### `learn_constant`, `learn_initial`, `learn_trajectory`
Learned values. `learn_trajectory` is the fourth mutually exclusive
initialization mechanism, owns all timesteps including t=0.
**Status: settled.**

### `dt` as a normal quantity
Not magic. Can be constant or per-step.
**Status: settled.**

### Multi-binding compilation
Same model, N binding sets, shared controller. Essential for sparse multi-study
training.
**Status: settled.**

## Compiler Analysis

### SCC detection
Automatic algebraic loop discovery. No solve blocks.
**Status: settled.**

### Hierarchical SCC decomposition
When `deriv(A, g_s)` feeds back into its own SCC: inner physics Newton + IFT
boundary + outer root-find over control variables.
**Status: settled.**

### Residual graph
Core semantic object. Factor graph with variable nodes, derived nodes, residual
factors, slot nodes.
**Status: settled.**

### Four-way component classification
Computational redundancy, square implicit, underdetermined residual,
overconstrained residual.
**Status: settled.**

### Closure policies
For overconstrained systems. Explicit approximations that relax the system.
**Status: settled.** Open: semantic interface for what a policy receives
(candidates, enumeration) is in deferred_review_findings.md.

### Plan inspection
Compilation plan as inspectable artifact. Shows SCCs, solver strategies,
symbolic resolutions, numerical fallbacks.
**Status: settled.**

### `#[verified_externally]`
Escape hatch for properties the compiler can't prove.
**Status: settled.**

### No proc macros

## NEW: Domains

### Domain as type annotation
A domain is an annotation on a type, not a separate concept. A type CAN be a
domain: `type Pasture : Domain<Euclidean, 2>`. This means the type's instances
are regions of space with a metric tensor field. Subdomains are domain types
contained within other domain types, handled by the normal containment tree.
Space IS a domain — there is no separate "space" concept.
**Status: new, syntax settled.** Open questions:
- Can users define custom metrics (curved spaces, non-Euclidean geometries)?
  Concept says yes, syntax undesigned.
- Coordinate transforms (polar vs cartesian) — are these on the domain type
  or on the instance?

### Coordinate naming with unit types — `as (x: Scalar<m>, y: Scalar<m>)`
Declared on the domain type definition. Coordinates carry units. Shared by all
fields over that domain. Example:
```
type Pasture : Domain<Euclidean, 2> as (px: Scalar<m>, py: Scalar<m>) { ... }
```
**Status: new, settled.**

### Domain extents are regular variables
Domain bounds are not const generics — they're regular variables that can be
set from data or even change during simulation. Declared inline with the
coordinate naming:
```
type Horse : Domain<Euclidean, 1> as (b: Scalar<m> in 0 m .. body_length) { ... }
```
The `in low .. high` syntax specifies the extent. `body_length` is a regular
field on the type. This means a horse's body length can be set per-instance
and can grow over time.
**Status: new, settled.**

### Boundary conditions reference coordinate bounds
Boundary conditions pin field values or derivatives at the endpoints of domain
extents:
```
boundary b = 0 m: body_temp = nose_temp
boundary b = body_length: -k * grad(body_temp) = h * (body_temp - ambient_temp)
```
The left side references the coordinate name and a bound value. The right side
is the condition.
**Status: new, settled.** Open questions:
- How do 2D/3D boundary conditions work? Named edges? Geometric selectors?
- How do irregular or curved boundaries work?
- How do boundary conditions interact with subdomain boundaries?

### Standard library domains
Interval, Grid2D, Grid3D, Network (branching 1D manifold), Surface. Shipped as
std lib, not hardcoded. Users can define custom domain types.
**Status: new, concept settled.** Open: the general syntax for defining a custom
domain type is undesigned.

### Discretization
Compiler chooses by default, user can override.
**Status: new, concept settled.** Open questions:
- Is override a Python-side config, a generic parameter, an annotation, or
  something else?
- Can the user hint resolution via data density (observation-driven
  discretization)?
- What discretization schemes are available? Finite difference, finite volume,
  spectral? Is this a std lib concern?

## NEW: Fields

### `field name: Type` — implicit over containing domain
Inside a domain type, `field temp: Temperature` is implicitly over that domain.
The `over` keyword is only needed for subdimensional fields — fields that vary
over fewer dimensions than the containing domain:
```
field nitrogen: Scalar<g_m2> over z    // 1D field in a 2D domain
```
Scalars are degenerate fields (a field sampled at one point). The compiler
understands this internally.
**Status: new, syntax settled.** Semantics settled: fields lower to arrays on a
mesh after discretization. Spatial operators derive from the domain's metric.

### Field sampling — `.at(x, y)`
Read a field's value at specific coordinates. Used for coupling discrete
entities to fields.
**Status: new, concept settled.** Open questions:
- Is `.at()` the right syntax?
- What about interpolation between mesh points after discretization?
- Does sampling a field at a point return an exact value or an interpolated
  approximation? (Probably the latter, but should the user know?)

### Spatial operators — `grad()`, `diverg()`, `laplacian()`, `curl()`
Derived from the domain's metric tensor. `grad(scalar_field) -> vector_field`,
`diverg(vector_field) -> scalar_field`, etc. `diverg` (not `div`) to avoid
confusion with division — final name settled.

Unit system extends through domains: spatial operators derive dimensional
signatures from coordinate units. `grad` divides by coordinate dimension,
`integrate` multiplies. The compiler infers this from the coordinate type
declarations. No new user-facing syntax needed.
**Status: new, concept settled.** Open questions:
- Full set of operators and their dimensional signatures (catalogue needed).
- How do operators work on subdomains vs parent domains?
- Do operators work on fields over Network domains (branching 1D manifolds)?
  If so, what does `grad` mean at a junction?

## NEW: Dynamic Topology

### `rule name: InputTypes -> OutputTypes { match/emit }`
Graph rewriting rules. Topology-level functions. Inputs are consumed from the
graph, outputs are produced. Drop is implicit in the bookkeeping — if two go in
and one comes out, one was consumed.
**Status: new, concept settled.** Open questions:
- Match syntax is too magical. Riley flagged this. Selection logic (which fish
  eats which) should be a function or explicit mechanism, not pattern matching.
  Needs redesign.
- Emit syntax — how are output instances created? Field-by-field assignment?
- Where do rules live? On the container type? Module scope?
- Can rules be generic?
- Can rules span multiple container types?

### `{ conserved }` — field annotation
Replaces the earlier separate `conservation` block concept. Annotated directly
on the field declaration:
```
type Fish {
    mass: Scalar<kg> { conserved }
}
```
Compiler checks that every rule consuming/producing instances of this type
accounts for fields marked `{ conserved }`. If a rule destroys a Fish and
doesn't route its mass somewhere, compile error.
**Status: new, mostly settled.** Open questions:
- Scoped conservation? `{ conserved within Pond }` for quantities that can
  leave the system via declared boundary fluxes?
- How do boundary fluxes interact with conservation? (Birds flying away with
  mass — is that a boundary condition on the container?)
- Does `{ conserved }` work for fields (continuous) as well as scalars?

### Two-tier collection sizing: `val` and `dyn`
Collections have two size modes:
- `[Horse; N_HORSE]` where `N_HORSE: val` — size fixed at instantiation, known
  to the compiler. Enables static Jacobian sizing.
- `[Fish; dyn]` — size changes at runtime as rules fire. Dynamic topology.

`dyn` replaces the placeholder `?` syntax. The keyword signals that this
collection participates in dynamic topology (birth/death/rules).
**Status: new, settled.** Open questions:
- How does `for i in fish` iteration work when the size is runtime-variable?
- Memory allocation strategy — preallocated max? Dynamic growth?
- How does the compiler handle Jacobian resizing when collection size changes?

### Rule conflict resolution
When two rules both want to consume the same instance in the same timestep.
**Status: new, identified as a problem, no design.** Options:
- Priority ordering (rules have explicit precedence)
- Mutual exclusion (compiler detects conflicts, forces user to specify)
- User-defined scheduling function
- Stochastic resolution (random winner)

### Rule firing conditions
Conditions under which rules trigger. Currently written as `when predicate`.
**Status: new, minimal design.** Open questions:
- Deterministic thresholds only? (Aligns with Riley's "stochasticity is
  observational" philosophy.)
- Can conditions reference quantities on both the match targets and the
  container?
- Rate-based firing for quasi-stochastic behavior? (Deferred per design
  decision — stochasticity is a workflow/Python concern, not a language
  concern.)

## NEW: Coupling & Kernels

### Kernel coupling — `K(a, b) -> strength`
Soft spatial coupling between two domains or between a domain and discrete
entities. At every point in domain A, coupling to domain B is weighted by the
kernel function. Replaces crisp wiring with graded, distance-weighted
interaction. Universal interface for root-soil, trunk-canopy, shading,
competition, dispersal.
**Status: new, concept settled.** Open questions:
- Is a kernel just a function used inside an `integrate` call, or does it need
  its own declaration? (See coupling-as-sugar section below.)
- Can kernels be learned (neural slots)? Concept says yes, syntax undesigned.
- How does kernel sparsity (characteristic length scale) get communicated to
  the compiler for spatial indexing optimization?

### Coupling may be sugar for relation with kernel integration
A coupling kernel might not need its own keyword. It could be syntactic sugar
for a relation that involves integration weighted by a kernel function:
```
relation root_uptake for i in 0..N_ROOT:
    roots[i].water_supply = integrate(
        K_root(roots[i], soil) * soil.water_potential,
        over = soil
    )
```
The compiler recognizes kernel-weighted integrals as a pattern and applies
spatial optimizations (characteristic length scale → skip distant interactions).
**Status: new, open.** Needs more design work to determine whether coupling
deserves its own keyword or is just a pattern the compiler detects.

### `integrate(field, over = entity)` for domain types
When a type is a domain, you can integrate a field over it:
```
total_heat = integrate(horses[i].metabolic_rate, over = horses[i])
```
This works because `horses[i]` is a domain type with known extent and metric.
The compiler handles the quadrature. This is how emergent properties (total
mass, total heat output, etc.) are computed from fields.
**Status: new, settled.**

## NEW: Cross-Scale Visibility

### Downward-only visibility
Parents see children, children never see up. A Pasture can reference
`horses[i].body_temp`, but a Horse cannot reference `pasture.air_temp` directly.
Information flows down through explicit relations on the parent:
```
relation local_climate for i in 0..N_HORSE:
    horses[i].ambient_temp = air_temp.at(horses[i].x, horses[i].y)
```
This is composition over inheritance. Children are self-contained components
that receive their environment as input — they don't know who owns them.
**Status: new, settled.**

### Emergent properties are downward aggregations
Emergent properties (whole-organism metabolic rate, total canopy LAI, etc.) are
computed by the parent integrating or summing over its children:
```
total_metabolic = integrate(horses[i].metabolic_rate, over = horses[i])
pasture_lai = sum(i in 0..N_HORSE, horses[i].lai)
```
No upward references needed. The parent is the one who cares about the
aggregate, so the parent computes it.
**Status: new, settled.**

### No `physical` keyword
Entities in a spatial domain don't need a special `physical` marker. They just
have scalar coordinate fields:
```
type Fly {
    pos: Scalar<m>        // position along the horse's body axis
    skin_temp: Temperature
}
```
The compiler doesn't need to know an entity is "physical" — it just sees fields
and relations. Spatial coupling happens through explicit `.at()` sampling and
relations, not through a type-level annotation.
**Status: new, settled.**

## NEW: Solver Architecture (implementation, not language)

### Block-decomposed solver
Intra-entity SCCs vmapped on GPU. Inter-entity coupling through shared resources
is sparse and block-decomposable. Block Gauss-Seidel outer iteration.
**Status: concept settled.** This is an implementation detail, not language
syntax. Included here because language design choices (fields, kernels, dynamic
topology) constrain the solver architecture.

### Static/dynamic classification
Compiler mechanically classifies model modules. If the type graph contains
`Option`, runtime-sized collections, or rules: dynamic topology (solver factory).
Otherwise: fully static (fixed factor graph, everything proven at compile time).
Superset, not replacement.
**Status: new, settled.**

### Performance targets
~3,000-3,400 tree-years/second on H100 for Sperry-scale models. Hectare-century
in ~17s forward, ~3min training. Memory-bound around ~10M entities.
**Status: back-of-envelope estimates.** Need validation once implementation
exists.

## Worked Example: Fly on a Horse in a Pasture

This example exercises domains, fields, spatial operators, cross-scale coupling,
subdimensional fields, boundary conditions, and discrete entities with scalar
positions. Three nested scales: pasture (2D) → horse (1D body axis) → fly
(scalar).

```myco
type Pasture<N_HORSE: val> : Domain<Euclidean, 2>
    as (px: Scalar<m>, py: Scalar<m>)
{
    field air_temp: Temperature
    field wind_speed: Scalar<m_per_s>

    horses: [Horse; N_HORSE]

    relation local_climate for i in 0..N_HORSE:
        horses[i].ambient_temp = air_temp.at(horses[i].x, horses[i].y)
        horses[i].ambient_wind = wind_speed.at(horses[i].x, horses[i].y)
}

type Horse<N_FLY: val> : Domain<Euclidean, 1>
    as (b: Scalar<m> in 0 m .. body_length)
{
    // position in parent domain
    x: Scalar<m>
    y: Scalar<m>

    // physical parameters
    body_length: Scalar<m>
    k: Scalar<W_per_m_K>
    h: Scalar<W_per_m2_K>

    // environment (pushed down by parent)
    ambient_temp: Temperature
    ambient_wind: Scalar<m_per_s>

    // fields over the body axis
    field body_temp: Temperature
    field metabolic_rate: Scalar<W_per_m3>

    // heat equation along the body
    temporal thermoregulation:
        d(body_temp) = k * laplacian(body_temp)
            + metabolic_rate
            - h * (body_temp - ambient_temp)

    // boundary conditions at body endpoints
    boundary b = 0 m:
        body_temp = nose_temp
    boundary b = body_length:
        -k * grad(body_temp) = h * (body_temp - ambient_temp)

    // discrete entities living on this domain
    flies: [Fly; N_FLY]

    relation fly_environment for i in 0..N_FLY:
        flies[i].skin_temp = body_temp.at(flies[i].pos)
}

type Fly {
    pos: Scalar<m>              // position along horse body axis
    skin_temp: Temperature      // pushed down by parent
    universal comfort_midpoint: Temperature = 310 K
    universal comfort_steepness: Scalar<1_per_K> = 0.5

    relation comfort:
        happiness = 1.0 / (1.0 + exp(comfort_steepness * (skin_temp - comfort_midpoint)))
}
```

**What this demonstrates:**
- **Domain as type annotation**: Pasture is a 2D Euclidean domain, Horse is a
  1D domain (body axis). Fly is not a domain — just a discrete entity.
- **Coordinate naming with units**: `as (px: Scalar<m>, py: Scalar<m>)`.
- **Domain extents as variables**: `in 0 m .. body_length` — not const generics.
- **Implicit field over**: `field body_temp: Temperature` is implicitly over the
  horse's body axis.
- **Cross-scale coupling via `.at()`**: `air_temp.at(horses[i].x, horses[i].y)`
  samples the pasture field at a horse's position.
- **Downward-only visibility**: Pasture pushes climate to horses, horses push
  skin temp to flies. No upward references.
- **Scalar coordinates, no `physical` keyword**: Fly has `pos: Scalar<m>`, not
  `physical position: Point over parent`.
- **Value parameters**: `N_HORSE: val`, `N_FLY: val`.
- **Boundary conditions**: Referencing coordinate bounds directly.
- **Spatial operators**: `laplacian(body_temp)`, `grad(body_temp)` — derived from
  the 1D domain metric.

## Deferred / Out of Scope

### PDE solvers
Field-first design means the type system and spatial operators are designed now.
Actual PDE solver backends (multigrid, spectral elements, etc.) are deferred.
One simple solver (finite differences on regular grid) to test the interface.

### Recursive types
`type Branch { children: [Branch; ?] }`. Deferred — dynamic topology with
rules may subsume the need. Revisit if real users find the flat-array
representation painful.

### Stochastic language primitives
Per design decision: the world is deterministic, stochasticity is an
observational bandwidth limit. Neural slots learn hidden deterministic rules.
Noise injection is a Python-side workflow concern. No `random()` in .myco.

### Proc macros
Deferred beyond v2.

### Full E3SM representation
Dynamical cores (atmosphere, ocean) are PDE-on-mesh problems. Myco can represent
the land surface, parameterizations, and biogeochemistry. Dynamical cores
require the deferred PDE solver extension.
