no inline instantiation whatsoever. nuke that from the spec. one way to do things. this is an antipattern.

nuke myco.load("lib.myco", root=..., params=...) inline instantiation (section 1.1, lines 98-111). if you want a model, write a model module. no sugar.

no bare magic numbers in .myco. every literal must be attached to a named declaration: `universal` (physics constant, valued inline, overridable from python) or `param` (value comes from workflow). dimensionless integers (0, 1, 2) in mathematical structure are the only exception. compiler rejects bare dimensioned literals in relations. python side can enumerate and override universals, must bind params.

node vs type terminology is confused. intent: `type` defines a structural pattern (like a struct), `node` is an instance of a type on the graph. currently the spec uses `node` for both definitions and instances. `type` is only used for scalar newtypes. fix: use `type` for all composite definitions too, reserve `node` for instances. big terminological change across the whole spec. CONFIRMED: `type Stem { ... }` for definitions, `node my_tree: Stem` for instantiation. library modules are all `type`, model module is where `node` appears.

nuke `root`. it's a causal concept in an acausal language. the compiler can identify connected components from the instantiated nodes and relations in a model module. no privileged entry point needed. `myco.load()` returns whatever graphs the module contains. if one, you get it directly. if multiple disconnected graphs, you get the collection. this also enables multi-model modules (e.g. calibration + prediction sharing structure).
