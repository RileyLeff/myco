- [x] treat as greenfield. remove v1 references, make v2 standalone.

- [x] no inline instantiation. nuke myco.load("lib.myco", root=..., params=...). one way to do things.

- [x] no bare magic numbers in .myco. every literal must be attached to a `universal` (physics constant, valued inline, overridable from python) or `param` (value comes from workflow). dimensionless integers (0, 1, 2) in mathematical structure are the only exception. compiler rejects bare dimensioned literals in relations. python side can enumerate and override universals, must bind params.

- [x] node vs type terminology: `type` defines a structural pattern (like a struct), `node` is an instance of a type on the graph. `type Stem { ... }` for definitions, `node my_tree: Stem` for instantiation. library modules are all `type`, model module is where `node` appears.

- [x] nuke `root`. it's a causal concept in an acausal language. the compiler identifies connected components from the instantiated nodes and relations in a model module. no privileged entry point. `myco.load()` returns whatever graphs the module contains.

- [ ] for library vs model module, a helpful analogy might be a rust source with or without a pub fn main(). it's still rust, compiling it just means something different. consider adding this framing to the spec.

- [ ] named generic arguments at instantiation sites. all positional or all named, no mixing. `SperryTree<V = WeibullVC, P = FarquharC3, N_SOIL = 4, N_CANOPY = 2>`. names come from the type definition. positional still valid for short signatures.

- [ ] OPEN DESIGN QUESTION: runtime dynamic graph topology. the fixed-factor-graph assumption needs to be replaced. a valid myco world description defines a space of possible graph structures, not a single fixed graph. the concrete topology depends on initialization data and changes during simulation (organisms born, die, grow new structure). this is first-class, not a future extension. implications: variable-size jacobians across timesteps, topology-aware temporal equations, incremental or re-run SCC decomposition, training over variable-structure graphs. needs dedicated design work. IMPORTANT: dynamic topology is a superset, not a replacement. the compiler classifies each model module mechanically — if the type graph contains Option, runtime-sized collections, or birth/death markers, it's dynamic (solver factory + runtime assertions). otherwise it's fully static (everything proven at compile time, fixed factor graph). no user annotation needed. static models compile to exactly the same thing as the current spec describes.
